## 한 화면 요약

- 전체 방향(단일 세대 모듈 보존 + 래퍼 + 흐름 서비스 + 콘텐츠 분리)은 결정 007-010, PRD와 정합. PRD 위반 또는 결정 보류 침범은 없다.
- **가장 큰 합류 계약 누락**: `FamilyName`/`FounderName`/`IntroLine`이 현재 `RunState`에 있다. `FamilyRunState`가 같은 필드를 또 가지면 데이터 진실의 출처가 둘이 된다. **`RunState`에서 가문 단위 필드를 분리해 `RunState`는 "한 세대 사건 진행 상태"로 좁히는 것**이 가장 작은 변경으로 일관성을 만든다. 1세대→2세대 진입 시 새 `RunState`를 만들 때마다 가문명을 복사할 필요가 없다.
- **두 번째로 큰 누락**: 후계 메커니즘의 입력으로 "큰 사건 결과"가 결정 008에 못 박혀 있지만, 현재 모델에 큰 사건 개념이 없다. `LoopEvent`/`EventChoice` 어디에 어떤 태그가 붙는지가 합류 계약이며 이번 단계에서 정하지 않으면 후계 로직이 가짜 입력으로 돌아간다.
- **세 번째로 큰 누락**: `--smoke`가 "최소 2세대 + 멸문"만 검증하면 **후계가 있는 경로**(다음 세대로의 정상 진입)는 자동으로 검증되지 않는다. 멸문은 후계 부재의 결과이므로, "후계 선택 → 다음 세대 진입 → 그 다음 세대에서 멸문"이 한 스모크에 들어가야 한다.
- 그 외: 종료 유형 enum 5종을 실제로 발화시키려는 시도, 상태 거동 매핑의 보관 위치, 임시값 위생 작업의 커밋 단위, 4트랙 병렬 분해의 합류 계약 표면은 아래 본문에서 다룬다.

## 주요 검토 의견

### 1. `RunState`를 감싸지 말고 좁혀라 — 가장 안전한 최소 변경

리뷰 요청서 §2와 §리뷰 요청 사항 1번은 "RunState 리네임 없이 FamilyRunState로 감싼다"를 가장 안전한 방향으로 본다. **방향은 맞지만, "감싸기"만으로는 데이터 중복이 발생한다.**

현재 `RunState`의 필드는 다음과 같다(`scripts/MvpLoopModels.cs:75-95`).

- `FamilyName` (가문 단위)
- `FounderName` (한 세대 인물 단위 — 사실은 "이 세대 인물명")
- `IntroLine` (한 세대 도입 문장 — 사실은 "이 세대 도입 문장")
- `CurrentEventIndex` (한 세대 진행)
- `Stats` (가문 단위 — 다세대 거동을 적용받는다)
- `Chronicle` (한 세대 항목식 기록)

가문 단위와 한 세대 단위가 섞여 있다. `FamilyRunState`가 위에 얹히고 동일 필드를 또 들면, 1세대→2세대 진입 시 어느 쪽이 진실의 출처인지가 모호해진다.

권장 변경은 두 가지 중 하나다.

- **권장 A — 좁히기**: `RunState`에서 `FamilyName`을 빼고 `FounderName`/`IntroLine`을 "현재 세대 인물명/도입 문장"으로 의미만 재해석한다. `Stats`도 사실은 가문 단위 상태이므로 `FamilyRunState`로 옮기고, `RunState`는 한 세대 진행 시 참조한다. `RunState`의 진짜 책임은 `CurrentEventIndex` + `Chronicle`(이 세대 항목식 기록)로 축소.
- **권장 B — 책임 분리만 명시(파일 변경 최소)**: `RunState`는 그대로 두되, "다세대 단계에서는 `FamilyName`을 **읽기 전용 카피**로 간주, 진실의 출처는 `FamilyRunState.FamilyName`"으로 합류 계약에 못 박는다. 단점: 코드만 보면 어느 쪽이 진실의 출처인지 안 보인다. 향후 세션이 `RunState.FamilyName`을 수정하는 함정이 남는다.

**Claude 권고는 권장 A.** `RunState`는 리네임하지 않더라도 필드 셋은 줄여야 한다. 이게 진짜 "리네임 없이 감싸기"가 의미를 갖는 형태다. 리뷰 요청서가 말한 "큰 리네임을 피한다"의 의도(diff 폭발 방지)는 이 안으로도 충족된다 — 이름은 그대로, 책임만 좁힌다.

이 결정은 §4(가문 상태가 `RunState`인지 `FamilyRunState`인지)와 §5(누적 연대기 보관 위치)에도 영향을 준다. 합류 계약으로 사전 확정 필요.

### 2. "큰 사건" 개념이 모델에 없다 — 결정 008/009의 본질 요구를 충족하려면 합류 계약 필수

리뷰 요청서 §7은 후계 메커니즘의 입력으로 "가문 상태, 이번 세대의 큰 사건 결과, 검증용 랜덤 요소"를 든다. 결정 008 §"세대 간 이어짐 매개"도 "큰 사건 결과"를 이어짐 매개로 못 박는다. 결정 009 §"정보량 차등"도 큰 사건과 보조 사건을 다르게 기록하라고 요구한다.

그런데 현재 모델(`MvpLoopModels.cs`)의 `LoopEvent`/`EventChoice`/`ChronicleEntry`에는 큰 사건 표식이 없다. 리뷰 요청서 §"수정 파일 후보"는 "**필요할 경우에만** `ChronicleEntry` 또는 `LoopEvent`에 큰 사건 여부/결과 태그를 추가"라고 두는데, 결정 008과 009 양쪽이 큰 사건을 본질 입력으로 두는 이상 **이 추가는 "필요할 경우"가 아니라 본 단계의 합류 계약**이다.

세 후보 위치 중 하나로 명시 권장:

- 후보 1: `LoopEvent.IsMajor`(bool) — 사건 단위로 큰/보조 구분. 단순. 후계 입력으로 쓰기 쉬움.
- 후보 2: `EventChoice.OutcomeTag`(string) — 선택지 단위로 결과 태그 부여. 같은 사건에서도 선택에 따라 큰 사건 결과가 달라지는 표현 가능. 후계 입력이 더 풍부.
- 후보 3: 둘 다 — 사건의 "큰 사건 여부" + 선택 결과의 "태그".

후보 2 또는 3 권장. 결정 008의 "큰 사건 결과"는 사건 자체가 아니라 사건의 결과(어떻게 매듭지어졌는가)이므로 선택지 단위 태그가 더 정확하다. 단 MVP 검증용이므로 태그 키 목록 자체는 자리표시자로(`tag-1`, `tag-2`) 두고 최종 키 어휘는 결정 보류로 남길 것.

이 합류 계약 없이 작업 A를 진행하면 `MultigenFlow`의 후계 함수가 입력 없는 함수로 시작했다가, 작업 B가 큰 사건 표식을 추가하면 시그니처 재작업이 발생한다.

### 3. 후계 메커니즘 — 혼합형 요구 충족. 단 "계산식 확정 오해 방지"의 코드 패턴이 더 명시적이어야 함

리뷰 요청서 §7과 §리뷰 요청 사항 3·4는 "혼합형 요구를 충족하면서도 계산식 확정으로 보이지 않는가"를 묻는다.

평가: 입력 셋(상태 + 큰 사건 결과 + 랜덤)은 결정 008의 혼합형 요구와 일치. 단 코드에 명시적 if/threshold가 들어가면 그 자체가 사실상 규칙으로 굳을 위험이 있다. PRD §9·결정 008은 후계 가능성 계산식을 결정 보류로 명시한다.

권장 코드 패턴:

- 후계 판정 함수 하나(`PlaceholderEvaluateSuccession`처럼 이름에 임시성이 드러나는 식별자)에 입력 셋을 모두 받고, 함수 본문 위에 `// MVP 검증용 임시 로직. 결정 008의 후계 가능성 계산식은 보류 상태. 본 함수는 다세대 흐름이 멸문/계승 양쪽 경로에 모두 도달 가능함을 확인하기 위한 자리표시자다.` 같은 코드 주석을 한 블록.
- 함수 내부에 매직 넘버 임계값을 두지 말고, 명백히 검증용임이 드러나는 형태(예: 후계 후보 수가 0이면 멸문, 1 이상이면 혼합형 선택, 큰 사건 결과 태그 중 특정 자리표시자 태그가 있으면 후보 0개로 강제 — 같은 식)로 두기.
- 임의 확률은 검증용 시드 고정 또는 비결정성 명시. 스모크 재현 가능성을 확보.

이렇게 두면 §리뷰 요청 사항 4("계산식 확정 오해 위험")가 코드 자체로 차단된다.

### 4. 세대 종결 유형 enum — 5종 정의는 OK. 단 1차 구현에서 발화하는 유형은 좁혀라

리뷰 요청서 §8은 `Extinction`/`BattleDeath`/`IllnessDeath`/`NaturalDeath`/`Deposed` 5종 enum을 제안한다. 결정 008 §"세대 종결 방식 다종성"은 다종성을 "원칙 수준에서 인정"으로 두고 별도 시스템을 요구하지 않는다.

평가: enum 정의 자체는 결정 008 범위 안. 단 5종을 **실제로 발화시키려면** 사건마다 "이 사건이 어떤 종결 유형을 만든다"를 결정해야 한다. 그러면 다음 결정 보류 항목들에 손이 들어간다.

- 어떤 사건이 전사·병사·자연사·폐위로 이어지는가
- 그 트리거가 상태 수치인가, 큰 사건 결과인가, 랜덤인가

권장: enum 5종은 정의하되, **MVP 검증용 자동 흐름이 실제로 발화시키는 유형은 2-3개로 한정**(`NaturalDeath`, `Extinction`, 그리고 가능하면 후계가 있는 경로 검증용으로 `Deposed` 또는 `BattleDeath` 중 하나). 나머지 유형은 enum 멤버로만 존재하고 `MultigenContent`의 자리표시자 요약 문장만 매핑. 발화 트리거는 검증용 자리표시자 로직.

이렇게 두면 enum이 다종성 자리를 확보하면서도 "각 종결 유형의 트리거 규칙"이라는 결정 보류 영역에 들어가지 않는다.

### 5. 상태 종류별 다세대 거동 — 매핑 보관 위치는 `MultigenFlow`가 더 자연스럽다

리뷰 요청서 §4는 선택지 A(`StateDefinition`에 `StateCarryKind` 추가) vs 선택지 B(`MultigenContent`에 상태 키별 다세대 거동 매핑) 중 B를 선호한다.

평가: 선택지 B는 결정 008의 "상태 이름·개수·종류 미확정"을 더 잘 보존한다는 점에서 동의. 단 매핑이 `MultigenContent`에 있는 것이 의미상 옳은지는 한 번 더 검토 필요.

- 상태의 **거동**(누적/감쇠/재구성)은 "콘텐츠"라기보다 "흐름 규칙"에 가깝다.
- `MultigenContent`에 두면 콘텐츠 작성자가 거동까지 같이 만진다 — 자리표시자 톤은 흩어지기 쉽다.
- `MultigenFlow`에 두면 흐름 서비스가 자기 입력 매핑을 가진 형태가 된다 — 코드 응집도 더 높다.

권장: **매핑은 `MultigenFlow`에**, 그리고 거동 적용 함수(누적/감쇠/재구성)도 같은 파일에 둔다. 콘텐츠 파일은 자리표시자 문장과 시드 데이터에 집중. 이 위치 결정만 다르고, 선택지 A vs B 큰 결정(StateDefinition을 안 건드린다)은 유지.

### 6. 상태 표시 계층화 — 결정 010의 "4-5단계 구간 라벨"이 계획에 빠져 있다

리뷰 요청서 §6은 기본 표시를 질적 구간 라벨로 두되 라벨 문구를 `구간 1`/`구간 2`로 자리표시자화한다고 한다. 결정 010 §"구간 라벨 단계 수"는 **4-5단계 권고**다.

평가: 라벨 문구가 자리표시자인 것은 결정 보류와 정합. 단 **단계 수가 몇이냐**는 결정 010의 권고 안에서 합류 계약으로 정해야 한다. 단계 수가 정해지지 않으면 `Stats[key] -> label` 매핑이 임의로 결정되어 사실상 임계값까지 만들게 된다.

권장: 4단계 또는 5단계 중 하나를 합류 계약으로 채택(권장: 4단계 — 첫 검증에서 단계 수 자체가 너무 많지 않게). 임계값은 검증용 균등 분할(예: -∞..−5 / −4..0 / 1..5 / 6..+∞)로 두고 `MultigenFlow`에 `PlaceholderStateBandLabel(key, value)` 같은 함수로 분리. **임계값과 단계 수의 최종 결정은 결정 보류**임을 코드 주석에 명시.

또한 결정 010의 "**세대 끝 서술 요약**"이 계획의 `GenerationRecord.SummaryParagraph`(후대 기록자 시점)와 같은 것인지, 별도 항목인지 합류 계약 필요. 리뷰 요청서 §6 "세대 끝 상태 요약"과 §5 "후대 기록자 시점 요약 문단"이 둘 다 등장하므로 두 항목인지 한 항목인지 분명하지 않다.

권장: **둘은 한 단락 안에 통합**. 세대 끝 후대 기록자 요약이 (a) 이번 세대의 의미, (b) 유산, (c) 다음 세대 상황 암시, (d) **세대 말 상태에 대한 서술적 요약**까지 포함한다. 별도 두면 정보 중복이 생긴다.

### 7. 연대기 문체 — 시점 분리는 OK. 단 결정 009의 다른 요구가 계획에서 약해져 있다

리뷰 요청서 §5는 진행 중 1인칭 + 세대 끝 후대 기록자 시점 분리를 다룬다. 결정 009와 일치.

다만 결정 009의 다음 요구가 리뷰 요청서에서 명시적으로 다뤄지지 않거나 약하게 다뤄진다.

- **정보량 차등**(큰 사건은 길고 화려하게, 보조 사건은 짧고 담백하게): §2(큰 사건 표식)와 묶이는 합류 계약. 큰 사건 표식이 들어가면 연대기 기록 길이/문체도 차등 적용 자리를 마련해야 한다. 1차 MVP에서 문체 차등을 완전히 구현할 필요는 없지만, **모델이 차등 자리를 가지는지** 여부는 정해야 한다.
- **후대 기록자 톤의 일관된 서정·중립**: 자리표시자 문장도 이 톤을 흔들지 않는 형태(예: `[자리표시자: 후대 기록자 요약]`처럼 톤이 비어 있는 표기)여야 한다. `[자리표시자: 후대가 이 세대를 영광으로 기록한다]` 같은 톤 침범 자리표시자는 피해야 한다.
- **자체 연호(`가문력 N년`)**: 어디서 등장하나? 진행 중 1인칭 기록? 세대 요약 후대 기록자 문장? 둘 다? 합류 계약 필요. 리뷰 요청서 §5는 "필요하면 `가문력 N년` 또는 `[자리표시자: 가문력 N년]`"이라고 두는데, 다세대 MVP의 핵심 정서가 "세대를 잇는 시간 감각"이므로 최소 한 곳(세대 요약)에는 들어가는 것을 합류 계약으로 권장.

### 8. 특성 스텁 — MVP 범위 안. 단 적용 효과가 코드에 들어가지 않게 묶어둬야 함

리뷰 요청서 §9는 특성 스텁을 키+표시명만 가진 최소 형태로 두고, 0-1개 수준으로 인물에 붙인다고 한다. 결정 008 §"특성 시스템 도입 수준"의 "최소 스텁"과 일치.

위험: 특성이 "적용 효과를 갖기 시작하면"(예: 특성이 후계 가능성이나 상태 변화에 직접 영향) 결정 008·PRD §9의 "특성 상속 확률"·"특성 이름/개수/구체 목록"이 결정 보류 영역인데 그 영역에 들어가게 된다.

권장: **특성은 표시 전용**으로 명시. 세대 요약/다음 세대 시작 정보에 한 줄로 표시되되, `MultigenFlow`의 어떤 로직에도 입력으로 들어가지 않는다. 후계 후보 매력도, 상태 변환, 사건 결과 어디에도 특성이 영향을 주지 않는다. 이 합류 계약을 코드 주석으로 남길 것.

상속도 마찬가지로 표시 수준. 검증용 임시 로직이 "후계 후보가 부모 특성을 일정 확률로 받는다"를 구현하면 안 된다. 후계 후보 자리표시자에 자리표시자 특성을 0-1개 붙이는 정도가 최대.

### 9. 임시값 위생 작업 포함 — 함께 처리 OK. 단 커밋 단위는 분리 권장

리뷰 요청서 §10은 위생 작업을 다세대 MVP 구현에 포함하는 것을 제안한다. `docs/temp-value-hygiene-delegation-draft.md`는 본래 별도 위임 초안 상태.

평가: 다세대 MVP에서 새 자리표시자(인물명, 후계 후보명, 후대 요약 문장)가 더 늘어나므로 같은 자리표시자 규칙으로 일관 처리하는 것이 합리적. 별도 단계로 분리하면 자리표시자 규칙이 다세대 구현과 다른 시기에 적용되어 두 번 손이 들어간다.

위험: 위생 작업이 한 PR/한 구현 단계에 다 묶이면 변경 표면이 커지고 리뷰에서 "다세대 MVP의 진짜 변경"과 "자리표시자 치환"이 섞인다.

권장: **포함하되 커밋 단위로 분리**. 같은 구현 흐름 안에서 다음 순서를 권장.

1. 커밋 1: 기존 `MvpLoopContent.cs`의 가문명/초대 인물명/도입 문장/세대 마무리 문장 자리표시자화 (위생 작업만). 사건 문안은 유지.
2. 커밋 2: `MvpLoopModels.cs` 변경(`RunState` 책임 좁히기 + 큰 사건 표식 추가).
3. 커밋 3: `scripts/MultigenModels.cs`, `scripts/MultigenFlow.cs`, `scripts/MultigenContent.cs` 추가.
4. 커밋 4: `Main.cs` 다세대 화면 확장 + `--smoke` 확장.

이 순서는 `mvp-first-loop` 흐름의 한 PR 단위 변경 크기와 비슷한 결로 다세대 MVP를 끊는다. 리뷰 가능 표면 유지.

또한 위생 작업이 다세대 MVP 범위를 흐릴 위험은 §리뷰 요청 사항 9와 §멈춤 지점에 둘 다 등장하는데, 위 커밋 단위 분리가 그 위험을 차단한다.

### 10. `--smoke` 검증 기준 — 부족. 후계 정상 경로가 누락

리뷰 요청서 §검증 기준은 "최소 2세대 + 멸문 도달"을 자동 통과 기준으로 둔다.

**부족**. 결정 008은 "후계가 있으면 다음 세대로 이어지고, 후계가 없으면 멸문"의 양쪽 경로를 본질 요구로 둔다. 멸문 도달만 검증하면 "후계 선택 → 다음 세대 진입"이 자동으로 검증되지 않는다. 1세대 종료가 곧바로 멸문이면 다음 세대 화면/`RunState` 생성 코드가 한 번도 실행되지 않을 수 있다.

권장: 스모크 경로를 다음 셋 중 하나로 명시.

- **경로 A(권장)**: 1세대 종료에서 후계 존재 → 다음 세대 진입 → 2세대 진행 → 2세대 종료에서 후계 부재(자리표시자 강제) → 멸문 화면. 한 실행에 "후계 정상 경로"와 "멸문 종료"가 모두 들어간다.
- 경로 B: 두 번의 스모크 실행(`--smoke=succession`/`--smoke=extinction` 분기). 코드 표면이 커진다.
- 경로 C: 한 실행에 멸문만 도달. 후계 정상 경로 미검증. **권장 안 함.**

또한 `--smoke` 출력에 다음 줄이 모두 나와야 한다(리뷰 요청서 §검증 기준의 보강).

- 세대 번호 — 1, 2 모두
- 각 세대 시작 인물 — 1세대, 2세대 모두
- 각 세대 사건 결과와 상태 변화
- 1세대 종료 시 후계 후보 목록(자리표시자라도)
- 후계 선택 또는 확인 한 줄
- 2세대 진입 줄(이전 세대 누적 연대기/상태가 이어졌음을 보이는 한 줄)
- 2세대 종료 시 후계 부재 판정 + 멸문 판정 줄
- 마지막 줄 `SMOKE_OK`

이 출력이 있으면 자동 스모크만으로 결정 008의 양쪽 경로가 검증된다.

### 11. 병렬 작업 분해 — 4트랙은 합류 계약 없이는 충돌 표면이 더 커진다

리뷰 요청서 §병렬 작업 분해는 A(모델+흐름), B(콘텐츠+위생), C(UI), D(스모크+보고) 4트랙. 이전 `mvp-first-loop` 리뷰는 작업 B 흡수로 2트랙 권장이었다.

평가: 다세대 MVP는 단일 세대보다 합류 표면이 크다. 작업 A의 `MultigenFlow`가 외부에 노출하는 함수가 단일 세대보다 많다(새 가문 생성, 세대 종료 처리, 후계 산출, 다음 세대 생성, 멸문 판정 등 최소 5개).

작업 A↔C 간 합류 계약을 사전에 못 박지 않으면 작업 C가 가짜 stub로 진행되다가 재작업 발생. 이전 리뷰의 §3 "선행 의존성" 원칙이 그대로 적용된다.

권장:

- 합류 계약 표면을 **함수 시그니처 수준**으로 사전 확정해 최종 작업 지시서에 박을 것. 아래 §"구현 전 반드시 정해야 할 합류 계약"에 후보 시그니처 정리.
- 작업 B(콘텐츠+위생)는 작업 A의 타입과 거동 매핑 키가 정해진 후 시작. 그 전 동시 시작은 손해.
- 작업 D(스모크+보고)는 명확히 작업 A-C 통합 후 직렬. 이건 리뷰 요청서가 이미 명시.
- 작업 A를 두 트랙으로 더 쪼개지 말 것: `MultigenModels.cs`와 `MultigenFlow.cs`는 같은 호흡에 한 사람이 쓰는 게 자연스럽다(모델 시그니처와 흐름 함수 시그니처가 함께 합의되어야 함).

대안: 작업 B를 작업 C에 흡수(이전 리뷰와 같은 패턴) — `Main.cs`를 쓰는 사람이 자리표시자 콘텐츠도 같이 쓴다. 단 자리표시자 위생 작업은 §9의 커밋 단위 분리를 유지. 4트랙이 꼭 필요한 분할이 아니라면 3트랙(A: 모델·흐름, B: 콘텐츠+UI, D: 스모크·보고)이 합류 표면을 더 줄인다.

### 12. 파일 3개 분리 — 현재 규모에 과하지 않음. 단 `MultigenFlow`의 책임 경계만 좁히기

리뷰 요청서 §리뷰 요청 사항 2번: 새 파일 3개(`MultigenModels.cs`/`MultigenContent.cs`/`MultigenFlow.cs`) 분리가 현재 규모에 과한지.

평가: 단일 세대가 2파일(Models + Content)이고 거기에 흐름 서비스(`Flow`) 1개가 추가된다. 다세대는 흐름 책임이 단일 세대보다 명확히 분리되어야 하므로(세대 종료 후 후계 산출, 다음 세대 생성 등이 UI에서 잘려 나와야 함) `Flow` 파일은 정당화된다.

단 `MultigenFlow`가 모든 다세대 책임의 진공청소기가 되지 않도록 책임을 사전에 좁힐 것:

- ✅ 새 가문 실행 생성
- ✅ 세대 종료 시 `GenerationRecord` 생성
- ✅ 상태 종류별 다세대 거동 적용(누적/감쇠/재구성)
- ✅ 후계 후보 산출(검증용 자리표시자 로직)
- ✅ 후계 선택 후 다음 세대 `RunState` 생성
- ✅ 멸문 판정
- ❌ UI 노드 생성/조작 — 합류 계약에 못 박힌 대로 `Main.cs`만.
- ❌ 콘텐츠 문장 생성 — `MultigenContent`가 자리표시자 문장을 제공, `MultigenFlow`는 호출만.

이 경계가 흐려지면 `MultigenFlow`가 1000줄짜리 단일 책임 위반 클래스가 된다. 합류 계약에 책임 화이트리스트로 박을 것.

- `GenerationEndType` 5종의 영문 식별자가 식별자 그대로 stdout/UI에 노출되지 않도록 주의. 표시는 자리표시자 한국어 매핑을 통해서만.
- 후계 후보 자리표시자 인물명이 어디서 만들어지는가. `MultigenContent`에 두면 콘텐츠. 다음 세대 진입 시 `RunState` 새로 만들 때 그 이름을 어떻게 결정하나(고정 시퀀스? 인덱스 기반?). 합류 계약 필요.
- 특성 스텁의 자리표시자 키도 `trait-1`/`trait-2`처럼 톤이 비어 있는 표기로 둘 것. 결정 008의 "특성 이름·개수 미확정"을 침범하지 않도록.

## 단일 세대 모듈 유지/확장 방식 검토

### 유지

`scripts/MvpLoopModels.cs`의 모델 6종(`StateDefinition`/`StateDelta`/`EventChoice`/`LoopEvent`/`ChronicleEntry`/`RunState`)은 단일 세대 사건 진행의 기본 모델로 유지. 단 §1 권장에 따라 `RunState` 필드 셋은 좁힌다(`FamilyName` 제거, `Stats`는 `FamilyRunState`로 이동, 또는 `RunState`는 한 세대 진행자로 좁힘).

§1 권장 A는 "이름은 그대로, 책임만 좁힘". 큰 리네임은 없다. `RunState`라는 이름이 의미상 더 맞지 않게 되더라도(이젠 "한 세대 진행 상태"인데 이름은 `RunState`) 이번 단계에서 리네임 비용은 회수되지 않는다. 후속 단계에서 `GenerationRunState` 등으로의 리네임은 별도 위임으로 둘 수 있다.

## 구현 전 반드시 정해야 할 합류 계약

작업 A(모델+흐름) ↔ 작업 B(콘텐츠+위생) ↔ 작업 C(UI) 사이.

### 1. `RunState` 책임 좁히기 채택 여부

- 권장: 권장 A — `RunState`는 한 세대 진행자로 좁힘. `FamilyName` 제거, `Stats`는 `FamilyRunState`로 이동.
- 영향: `MvpLoopContent.CreateInitialRunState()` 시그니처 변경. `Main.cs`가 가문명을 `FamilyRunState`에서 읽도록 변경.

### 2. 큰 사건 표식 위치

- 권장: `EventChoice.OutcomeTag`(string, nullable). 사건 단위로는 `LoopEvent.IsMajor` 불필요(선택지 결과 단위로 충분).
- 영향: `MvpLoopModels.cs` 변경. `MvpLoopContent.Events`의 일부 선택지에 자리표시자 태그 추가. 후계 함수가 `FamilyRunState.CarriedMajorOutcomes`를 입력으로 받음.

### 3. 가문 상태 진실의 출처

- 권장: `FamilyRunState.FamilyStats` 한 곳. 한 세대 진행 중 변화는 `MultigenFlow`가 세대 종료 시 가문 단위로 모음.
- 대안: `RunState.Stats`를 그대로 유지하고 세대 종료 시 `FamilyRunState`로 옮기는 모델. 어느 쪽이든 단일 출처 원칙은 지킬 것.

### 4. 누적 연대기 보관 위치

- 권장: `RunState.Chronicle`(한 세대 항목식) + `FamilyRunState.History`(끝난 세대들의 `GenerationRecord` 누적). 전 가문 단위 항목식 누적 리스트는 두지 않음(세대별 그룹화가 자연스럽다).

### 5. 종료 유형 enum 멤버 vs 발화 유형

- 권장: enum 5종 모두 정의. 검증용 자동 흐름이 실제로 발화시키는 유형은 `NaturalDeath`(후계 있음 경로) + `Extinction`(후계 없음 경로) 2종으로 한정. 나머지 3종은 `MultigenContent`의 자리표시자 매핑만.

### 6. 상태 거동 매핑 위치

- 권장: `MultigenFlow`에 매핑(`Dictionary<string, StateCarryKind>` 또는 별도 enum). 콘텐츠 파일은 자리표시자 문장만.
- 매핑의 임시성은 코드 주석으로 명시.

### 7. 라벨 단계 수와 임계값 분할

- 권장: 4단계, 균등 분할(범위는 자리표시자 임계값). `MultigenFlow.PlaceholderStateBandLabel(key, value)` 함수로 분리.

### 8. 세대 끝 후대 요약 단락의 포함 항목

- 권장: 한 단락 안에 (a) 이번 세대 의미, (b) 유산, (c) 다음 세대 상황 암시, (d) 세대 말 상태 서술 요약을 통합. `GenerationRecord.SummaryParagraph` 한 필드.

### 9. 자체 연호(`가문력 N년`) 등장 위치

- 권장: 최소 한 곳 — 세대 요약 헤더. 진행 중 기록은 결정 009 §"시간 표시" 결정 보류 영역으로 두고 추가하지 않음. 표기는 자리표시자(`[자리표시자: 가문력 N년]` 또는 `가문력 N년`) 중 후자 권장(결정 009가 예시로 명시).

### 10. UI 진입점 함수 화이트리스트

- `Main.cs`는 `MultigenFlow`의 §"흐름 진입점 후보" 4개 public 함수만 호출.
- `Main.cs`는 `MultigenContent`의 자리표시자 시드 함수만 호출(가문/인물명 시드, 후계 후보 시드, 종료 유형 자리표시자 문장, 후대 요약 템플릿).
- 그 외 진입점은 합류 계약 위반.

### 11. 자리표시자 톤 규칙

- 인물명/가문명/사건명/후계 후보명: 자리표시자형(`가문 A`, `인물 1`, `[자리표시자: 도입 문장]`).
- 후대 기록자 요약: 톤이 비어 있는 자리표시자(`[자리표시자: 후대 기록자 요약]`). 영광/오욕 같은 평가 어휘 금지.
- 종료 유형 표시 문장: 자리표시자(`[자리표시자: 자연사 요약]`).
- 사건 문안/연대기 본문: 기존 `MvpLoopContent.Events`의 한국어 문안 유지(이번 범위 밖, 리뷰 요청서 §10과 일치).

### 12. 커밋 단위 분리

- §9의 4단계 커밋 순서를 최종 작업 지시서에 명시.

## 검증 방법 보완 제안

### 자동 스모크 — 경로 A로 확장

§10의 권장. 1세대(후계 있음, 자연사) → 2세대 진입 → 2세대(후계 없음, 멸문) → `SMOKE_OK` 한 실행에 양쪽 경로가 모두 들어간다.

스모크 stdout에 다음 줄이 모두 등장해야 한다.

```
[SMOKE] FAMILY name=가문 A
[SMOKE] GEN 1 character=인물 1 intro=...
[SMOKE] EVENT 1 ...
[SMOKE] DELTA ...
[SMOKE] CHRONICLE ...
... (사건 3개 반복)
[SMOKE] GEN_END type=NaturalDeath summary=...
[SMOKE] SUCCESSION candidates=2
[SMOKE] SUCCESSION_PICK name=인물 2
[SMOKE] GEN 2 character=인물 2 carried_stats=... carried_chronicle_count=...
[SMOKE] EVENT 1 ...
... (사건 3개 반복)
[SMOKE] GEN_END type=Extinction summary=...
[SMOKE] EXTINCT family=가문 A
SMOKE_OK
```

세대 번호, 시작 인물, 후계 판정, 다음 세대 진입, 멸문 판정 줄이 모두 등장하면 결정 008의 양쪽 경로가 자동 검증된다.

### 수동 GUI 1회

이전 리뷰와 같이 사람이 한 번 직접 확인. 결정 009의 "후대 기록자 톤", 결정 010의 "정보 계층화 감각"은 stdout만으로는 검증 불가. 수동 1회 확인을 보고 형식 필수 항목으로 유지.

수동 확인 관찰 기준은 리뷰 요청서 §검증 기준에 이미 6개로 정리됨. 한 가지만 보강: **2세대 진입 시점에 이전 세대 연대기/상태가 보이는지** 한 줄 확인을 명시. 결정 008의 "세대 간 이어짐 매개"가 화면에서 보이는지 직접 확인.

### `dotnet build` + `--headless --import`

이전 리뷰 그대로. 단 다세대 MVP는 새 C# 파일 3개가 추가되므로 `--import`가 반드시 한 번 필요(scripts/ 디렉터리에 새 파일이 생겼으니 Godot이 `.uid` 생성/스크립트 클래스 등록).

## Project Codex가 사용자에게 재확인해야 할 항목

대부분의 결정 보류 영역은 리뷰 요청서가 잘 보존했다. 다음 항목만 사용자(Root) 재확인 필요 가능성.

1. **§1 권장 A 채택 여부 — `RunState` 책임 좁히기**
   - 리뷰 요청서는 "리네임 없이 감싼다"라고 했으나, 좁히기를 추가로 받아들이면 `MvpLoopContent.CreateInitialRunState()`의 호출 흐름도 살짝 바뀐다. 단일 세대 모듈을 "그대로 보존" 정도가 어디까지인지 사용자 의도 재확인이 안전.
2. **§2 큰 사건 표식을 모델에 추가하는 결정**
   - 결정 008/009의 "큰 사건 결과"는 본질 요구. 모델에 표식이 들어가는 것은 결정 보류 침범이 아니라 결정 채택. 그래도 `MvpLoopModels.cs`의 단일 세대 모델을 한 번 더 건드린다는 점에서 사용자 의도 재확인 권장.
3. **§10 스모크 경로 A 채택 여부**
   - "최소 2세대 + 멸문"이라는 리뷰 요청서 기준을 "양쪽 경로 한 실행에" 기준으로 강화하는 변경. 검증 범위가 늘어남.
4. **§9 위생 작업 커밋 단위 분리**
   - 위생 작업을 한 다세대 MVP 흐름 안에 두되 커밋 단위는 분리. 별도 위임으로 유지할지, 같이 가되 분리 커밋으로 갈지 사용자 선호 재확인 권장.

나머지 합류 계약 사항(§3·§4·§5·§6·§7·§8·§11·§12)은 PRD/결정문서 영역 안에서 Project Codex가 판단해 최종 작업 지시서에 박을 수 있다고 본다.

- ❌ 상태 항목 이름/개수/수치/감쇠율 임시 수준 초과 — 모두 자리표시자 + 검증용으로 유지.
- ❌ 후계 가능성 계산식 최종화 — §3 권장으로 회피.
- ❌ 특성 상속 확률/목록 최종화 — §8 권장으로 회피.
- ❌ 사건 데이터 저장 형식 확정 — 기존 정적 배열 유지, 큰 사건 표식만 추가.
- ❌ 완성형 UI 구조/아트 스타일 결정 — 동적 UI 패턴 유지.
- ⚠️ 임시값 위생 작업이 다세대 MVP 구현 범위를 흐림 — §9 커밋 단위 분리로 회피(완전 회피 여부는 사용자 재확인 필요).
- ⚠️ 병렬 작업 분해가 충돌을 키움 — §11 3트랙 권장 + 합류 계약 사전 확정으로 회피.

**위 리뷰 권장을 반영한 최종 작업 지시서 작성으로 진행 가능.** 단 §"사용자 재확인" 4항은 Project Codex가 판단해 필요 시 Root 재확인.
