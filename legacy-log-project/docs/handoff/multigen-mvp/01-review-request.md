## 제품 범위

구현 계획은 다음 사용자 흐름을 만족해야 한다.

1. 플레이어는 새 실행을 시작한다.
2. 플레이어는 고정된 첫 가문과 초대 인물의 시작 정보를 본다.
3. 플레이어는 한 세대 안에서 단일 세대 사건 루프를 진행한다.
4. 선택 결과는 결과 문구, 최소 상태 변화, 진행 중 연대기 항목으로 피드백된다.
5. 세대 사건이 끝나면 세대 요약을 본다.
6. 세대 요약은 누적 연대기, 최종 상태 요약, 후대 기록자 시점의 요약 문장을 포함한다.
7. 세대 종료 이후 후계 또는 멸문 여부가 판단된다.
8. 후계가 있으면 플레이어는 다음 세대 진입을 확인하고 새 세대로 넘어간다.
9. 다음 세대는 이전 세대의 가문 상태, 큰 사건 결과, 연대기를 이어받는다.
10. 후계 부재 상태에서 현 세대 인물이 사망하거나 퇴장해 다음 세대로 이어질 수 없으면 가문 멸문을 확인한다.
11. MVP는 최소 2세대 이상을 관찰할 수 있어야 하며, 고정 상한 N세대 종료가 아니라 멸문까지 진행되는 구조여야 한다.

## 다세대 MVP 구현 계획

### 1. 단일 세대 루프 유지

`MvpLoopModels.cs`의 사건/선택/상태 변화/연대기 타입은 단일 세대 사건 루프의 기본 모델로 유지한다.

`RunState`는 현재 이름이 "실행 상태"처럼 보이지만 실제 책임은 한 세대 사건 진행에 가깝다. 이번 구현에서 큰 리네임을 하지 않고, 새 다세대 상태가 `RunState`를 현재 세대 루프로 포함하거나 감싼다. 리네임은 범위와 diff를 키우므로 이번 단계에서는 권장하지 않는다.

`MvpLoopContent.Events`의 3개 사건은 우선 각 세대에서 재사용 가능한 검증용 사건 세트로 둔다. 대량 사건 콘텐츠와 세대별 별도 사건 풀은 제외한다.

### 2. 다세대 상태 래퍼 추가

새 파일 `scripts/MultigenModels.cs`를 추가해 단일 세대 루프 위에 놓이는 실행 상태를 분리한다.

포함 후보:

- `FamilyRunState`
  - 가문 표시명
  - 현재 세대 번호
  - 현재 세대 인물 표시명
  - 현재 세대의 `RunState`
  - 전체 가문 연대기 목록
  - 이전 세대에서 남은 큰 사건 결과 목록
  - 현재 인물의 특성 스텁 목록
  - 멸문 여부
- `GenerationRecord`
  - 세대 번호
  - 인물 표시명
  - 세대 종료 유형
  - 세대 중 누적된 연대기 항목
  - 후대 기록자 시점 요약 문단
  - 다음 세대에 남는 유산 문장
- `GenerationEndResult`
  - 종료 유형: 멸문, 전사, 병사, 자연사, 강제 폐위 같은 표시 가능한 값
  - 현 세대 인물이 사망 또는 퇴장했는지 여부
  - 후계 후보 목록
  - 멸문 여부
- `SuccessionCandidate`
  - 후보 표시명
  - 후보 설명 문장
  - 특성 스텁 목록
  - 다음 세대 시작 상태에 적용할 임시 영향
- `TraitStub`
  - 키와 표시명 또는 설명만 가진 최소 스텁

이 타입들은 완성형 혈통/계보 시뮬레이션이 아니라 다세대 MVP가 "이어지는 자리"를 갖게 하는 최소 구조다.

### 3. 다세대 콘텐츠와 임시 로직 분리

새 파일 `scripts/MultigenContent.cs`를 추가해 다세대 전환에 필요한 임시 문구와 검증용 규칙을 둔다.

포함 후보:

- 세대별 자리표시자 인물명 생성
- 후계 후보 자리표시자 생성
- 세대 종료 유형별 자리표시자 요약 문장
- 세대 요약의 후대 기록자 시점 문장 템플릿
- 다음 세대가 직면할 상황 암시 문장
- 특성 스텁 후보
- 검증용 후계/멸문 판단 헬퍼

주의:

- 후계 가능성 계산식, 감쇠율, 특성 상속 확률은 확정하지 않는다.
- 필요한 경우 `MvpVerification` 또는 `Placeholder`라는 이름이 드러나는 헬퍼와 주석으로 임시 로직임을 명시한다.
- 검증용 로직은 상태, 큰 사건 결과, 랜덤 요소를 모두 입력으로 받을 수 있게 하되, 최종 확률/계산식처럼 보이지 않게 한다.

### 4. 세대 전환 서비스 추가

새 파일 `scripts/MultigenFlow.cs`를 추가해 다세대 진행 규칙을 UI에서 분리한다.

책임 후보:

- 새 가문 실행 생성
- 한 세대가 끝났을 때 `GenerationRecord` 생성
- 상태 종류별 다세대 거동 적용
  - 재산형: 누적
  - 평판형: 임시 감쇠 처리
  - 관계형: 부분 재구성
- 후계 후보 산출
- 후계 선택 후 다음 세대 `RunState` 생성
- 후계 부재와 사망/퇴장 조건이 겹칠 때 멸문 판정

상태 종류별 거동은 `StateDefinition`에 새 속성을 추가하거나 별도 매핑으로 둔다. 현재 `StateDefinition`이 `Key`와 `Label`만 가지므로, Project Claude 리뷰에서 다음 두 선택지를 검토받는다.

- 선택지 A: `StateDefinition`에 `StateCarryKind`를 추가한다.
- 선택지 B: `MultigenContent`에 상태 키별 다세대 거동 매핑을 둔다.

Project Codex의 현재 선호는 선택지 B다. 이유는 기존 단일 세대 모델을 덜 건드리고, 현재 상태 이름/개수/종류가 확정되지 않았음을 더 분명히 유지할 수 있기 때문이다.

### 5. 연대기 문체 분리

진행 중 기록은 현재 `EventChoice.ChronicleText`를 계속 사용하되, 문체가 1인칭 인물 본인 시점이 되도록 기존 임시 콘텐츠의 향후 수정 방향을 명시한다.

세대 종료 후 요약은 새 `GenerationRecord.SummaryParagraph` 또는 유사 필드로 분리한다. 이 요약은 후대 기록자 시점으로 작성하고 다음을 포함해야 한다.

- 이번 세대의 의미
- 이전 세대 또는 이번 세대의 유산
- 다음 세대가 직면할 상황 암시

최종 문안, 자체 연호 명칭, 시대/세계관 어휘는 확정하지 않는다. 시간 표시는 필요하면 `가문력 N년` 같은 결정문서의 예시 또는 `[자리표시자: 가문력 N년]` 형태로 둔다.

### 6. 상태 표시 계층화

`Main.cs`의 현재 상태 패널은 숫자만 보여준다. 다세대 MVP에서는 기본 표시와 상세 표시를 분리한다.

계획:

- 기본 상태 패널
  - 상태별 질적 구간 라벨을 보여준다.
  - 라벨 문구는 최종 문구가 아니므로 `구간 1`, `구간 2` 같은 자리표시자 또는 명확한 임시 라벨로 둔다.
- 세대 끝 상태 요약
  - 숫자 목록만이 아니라 세대 말 상태를 설명하는 짧은 서술 요약을 추가한다.
- 추가 정보 표시
  - 별도 토글 버튼 또는 간단한 접기/펼치기 영역으로 현재 숫자와 변화 방향을 보여준다.
  - 구체 UI 디자인은 확정하지 않고, 현재 코드 기반 동적 UI 안에서 확인 가능한 최소 형태로 둔다.

리뷰 요청 지점:

- 추가 정보창을 별도 팝업으로 만들면 UI 디자인 결정처럼 보일 수 있다. 따라서 Project Codex의 현재 선호는 `Main.cs` 안의 단순 토글 영역이다.

### 7. 후계 메커니즘 최소 구현

후계 메커니즘은 혼합형 요구를 만족해야 한다.

계획:

- 세대 종료 시 `MultigenFlow`가 다음 입력을 함께 본다.
  - 가문 상태
  - 이번 세대의 큰 사건 결과 또는 선택 결과 태그
  - 검증용 랜덤 요소
- 후계 후보가 있으면 플레이어에게 최소 선택권을 준다.
  - 후보 1명만 있으면 "다음 세대로" 확인 버튼만 둘 수 있다.
  - 후보 2명 이상이면 선택 버튼을 제공한다.
- 후계 후보가 없고 현 세대 인물이 사망 또는 퇴장한 종료 유형이면 멸문으로 간다.
- 후계 후보가 있으면 종료 유형이 전사, 병사, 자연사, 강제 폐위여도 다음 세대로 이어질 수 있다.

계산식과 확률은 확정하지 않는다. Project Claude 리뷰에서는 이 계획이 "혼합형" 요구를 충족하면서도 후계 가능성 계산식을 확정하지 않는지 확인받는다.

### 8. 세대 종결 방식 다종성

MVP에서는 세대 종결 방식의 다종성을 결과 유형과 요약 문장 수준에서만 구현한다.

계획:

- `GenerationEndType` 또는 유사 enum을 둔다.
- 표시 가능한 유형 후보:
  - `Extinction`
  - `BattleDeath`
  - `IllnessDeath`
  - `NaturalDeath`
  - `Deposed`
- 각 유형은 세대 요약 문장과 멸문 판단에만 사용한다.
- 각 유형마다 별도 시스템, 이벤트 풀, 상태 계산은 만들지 않는다.

종료 유형 후보의 최종 한국어 문구는 확정하지 않는다. 코드 enum 이름은 구현 식별자이고, 화면 표시 문구는 자리표시자로 둔다.

### 9. 특성 시스템 최소 스텁

특성은 완성형 유전/교육/랜덤 시스템이 아니라 "다음 세대 인물이 이전 세대와 완전히 무관하지 않다"는 자리만 제공한다.

계획:

- `TraitStub`는 키와 자리표시자 표시명만 가진다.
- 첫 세대 인물과 후계 후보에 0-1개 수준의 임시 특성을 붙일 수 있다.
- 세대 요약이나 다음 세대 시작 정보에 특성 스텁을 한 줄로 표시할 수 있다.
- 상속 확률, 교육 효과, 구체 특성 목록은 확정하지 않는다.

### 10. 임시값 위생 작업

`docs/temp-value-hygiene-delegation-draft.md`의 범위는 다세대 MVP 구현 계획에 포함하는 편이 낫다.

포함 이유:

- 다세대 MVP는 가문명, 인물명, 도입 문장, 세대 마무리 문장이 더 자주 노출된다.
- 현재 `하서가`, `하서 단` 같은 값이 제품 설정처럼 굳어질 위험이 있다.
- 이번 구현에서 새 세대명/후계 후보명까지 추가되면 임시값이 더 늘어난다.

계획:

- `scripts/MvpLoopContent.cs`의 가문명, 초대 인물명, 도입 문장, 기존 세대 마무리 문장을 자리표시자형 표현으로 바꾼다.
- 신규 다세대 콘텐츠도 같은 자리표시자 규칙을 따른다.
- 예시:
  - `가문 A`
  - `인물 1`
  - `인물 2`
  - `[자리표시자: 도입 문장]`
  - `[자리표시자: 세대 마무리 문장]`
  - `[자리표시자: 후대 기록자 요약]`
- 사건 문안 전체 교체는 이번 범위에 넣지 않는다. 사건 문안은 기존 단일 세대 검증 콘텐츠로 유지하되, 필요하면 주석으로 임시성을 더 명확히 한다.

Project Claude 리뷰에서 확인받을 점:

- 이 위생 작업을 다세대 MVP 구현과 함께 수행해도 범위가 과하지 않은지
- 가문명/인물명/도입/마무리 문장만 바꾸고 사건 문안은 유지하는 경계가 적절한지

.\scripts\godot.cmd --headless -- --smoke

## 리뷰 요청 사항

Project Claude는 다음 항목을 우선 검토한다.

1. 기존 단일 세대 `RunState`를 유지하고 `FamilyRunState`로 감싸는 방향이 현재 코드에 가장 작은 변경인지
2. `MultigenModels.cs`, `MultigenContent.cs`, `MultigenFlow.cs` 분리가 현재 규모에 과한지
3. 후계 메커니즘을 상태, 큰 사건 결과, 랜덤 요소 입력을 받는 검증용 로직으로 두는 것이 "혼합형" 요구를 충족하는지
4. 위 방식이 후계 가능성 계산식을 확정하는 것으로 오해될 위험이 있는지
5. 세대 종결 유형 enum을 두는 것이 결과 유형과 요약 문장 수준을 넘어서는지
6. 특성 스텁 모델이 MVP 범위 안에 머무르는지
7. 상태 종류별 다세대 거동을 `StateDefinition` 확장 대신 별도 매핑으로 두는 선호가 적절한지
8. 질적 라벨과 상세 숫자 토글을 `Main.cs` 동적 UI 안에 두는 것이 추가 정보창 요구를 최소 범위로 만족하는지
9. 임시값 위생 작업을 다세대 MVP 구현 범위에 포함하는 것이 적절한지
10. `--smoke`가 최소 2세대 이상과 멸문까지 자동 검증해야 한다는 기준이 충분한지
11. 병렬 작업 분해가 실제 파일 소유권 기준으로 충돌 없이 가능한지
12. 최종 구현 지시서 전에 더 좁혀야 할 합류 계약이 있는지

## 멈춤 지점

다음 판단이 필요하면 Project Claude는 구현으로 넘어가지 말고 리뷰 의견으로 돌려준다.

- PRD에 없는 게임 규칙을 확정해야 하는 경우
- 상태 항목 이름, 개수, 수치 범위, 감쇠율이 임시 검증 수준을 넘어서는 경우
- 후계 가능성 계산식 또는 확률을 최종 규칙처럼 정해야 하는 경우
- 특성 상속 확률 또는 특성 목록을 최종화해야 하는 경우
- 사건 데이터 저장 형식을 확정해야 하는 경우
- 완성형 UI 구조나 아트 스타일 결정을 요구하는 경우
- 임시값 위생 작업이 다세대 MVP 구현 범위를 흐린다고 판단되는 경우
- 병렬 작업 분해가 오히려 충돌을 키운다고 판단되는 경우

- 다세대 모델과 파일 분리 개선 제안

- 구현 전 반드시 정해야 할 합류 계약
- 검증 방법 보완 제안
- Project Codex가 사용자에게 재확인해야 할 항목

## Project Codex 보고 요약

### 계획 요약

현재 단일 세대 첫 루프 모듈은 유지하고, 그 위에 다세대 실행 상태와 세대 전환 흐름을 추가한다. 단일 세대 사건 3개는 한 세대 실행기로 재사용한다. 세대 종료 후에는 후계 후보 또는 멸문 여부를 판단하고, 후계가 있으면 다음 세대 `RunState`를 생성한다. 연대기는 진행 중 항목식 기록과 세대 끝 후대 기록자 요약으로 분리한다. 상태 표시는 기본 질적 라벨과 상세 숫자/변화 방향 토글로 계층화한다.

### 결정 보류로 남긴 항목

- 상태 값 이름, 개수, 최종 표시명
- 상태 수치 범위, 감쇠율, 구간 임계값
- 후계 가능성 계산식과 확률
- 특성 상속 확률과 최종 특성 목록
- 상태 구간 라벨의 실제 문구
- 추가 정보창의 최종 UI 레이아웃
- 자체 연호의 최종 명칭
- 시대/세계관 톤
- 가문/인물 이름 후보
- 사건과 세대 요약의 최종 콘텐츠 문안

### Project Claude에게 확인받을 질문

- 새 다세대 파일 3개 분리가 현재 규모에 적절한가
- `RunState`를 리네임하지 않고 감싸는 방향이 가장 안전한가
- 검증용 후계 로직이 혼합형 요구를 만족하면서도 계산식 확정으로 보이지 않는가
- 임시값 위생 작업을 이번 구현에 포함해도 범위가 흔들리지 않는가
- 다세대 `--smoke`의 통과 기준이 사용자 흐름 검증에 충분한가

### Root Codex 판단이 다시 필요한 항목

현재 계획 기준으로 즉시 Root Codex 재확인이 필요한 항목은 없다. 단, Project Claude 리뷰에서 후계 판단, 상태 라벨, 특성 스텁이 PRD 범위를 넘어선다고 판단되면 Root Codex에 다시 확인해야 한다.
