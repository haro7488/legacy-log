## 1. 주요 검토 의견

전반적으로 계획은 결정 010/011/012의 방향과 충돌하지 않으며, 새 게임 기능을 만들지 않고 정보 구조만 정리한다는 점에서 범위가 잘 잡혀 있다. 다만 다음 다섯 가지가 구현 지시서 확정 전에 해소돼야 회귀 위험을 낮출 수 있다.

1. **`유산` 탭의 실질 정보 부족**: 현재 자료로 보여줄 수 있는 정보가 `CarriedOutcomeTags`(자리표시자 문자열 `outcome-1` 반복)와 표시 전용 특성 스텁뿐이다. 1세대에서는 거의 빈 탭이 되고, 2세대에서도 `상태` 탭과 정보 출처가 겹친다. 탭 수를 3개로 줄이거나, `유산`을 `가문사` 탭의 상단 요약으로 흡수하는 안을 함께 검토할 필요가 있다.
2. **상세 토글 → `상태` 탭 흡수의 정보 계층화 약화**: 결정 010은 "캐주얼한 시선에는 4단계 라벨, 깊이 보고 싶은 사용자에게는 숫자/방향"이라는 계층화를 명시한다. 두 정보를 `상태` 탭 한 화면에 평탄하게 나열하면 이 의도가 깨진다. 탭 내부에 sub-section 분리 또는 토글 유지를 합류 계약으로 명문화해야 한다.
3. **`project.godot`의 stretch/window 설정 부재**: 현재 파일에는 `[display]` 섹션 자체가 없다. 작은 창과 세로형 검증을 제품 목적으로 잡았다면 stretch mode/viewport 기준 크기/resizable 설정 추가가 사실상 필수다. 리뷰 요청서가 "변경 시 UI/UX 검증 목적에 한정"이라고 적었지만, **변경 자체가 필요한가**의 판단을 사용자에게 재확인할 필요가 있다.
4. **사건 화면의 선택지 위치 모호성**: 계획서가 "ActionBar = 항상 보임"과 "사건 화면에서는 선택지 버튼을 세로 목록으로 보여준다"를 동시에 적었다. 선택지가 ActionBar에 들어가는지 MainContent에 들어가는지 명시되지 않았다. ActionBar에 넣으면 향후 선택지 수가 늘 때 본문이 밀린다. MainContent에 넣으면 작은 세로 화면에서 본문 끝까지 스크롤해야 선택지가 보인다. 결정이 필요하다.
5. **멸문 화면의 "다음 행동" 누락**: 현재 `ShowExtinction`은 버튼이 하나도 없다. 결정 011의 "다음에 눌러야 할 주요 행동이 명확한가"를 충족하려면 최소한 "게임이 끝났습니다" 같은 비활성 상태 표시 또는 disabled 상태의 라벨이 ActionBar에 들어가야 한다. 재시작 버튼은 새 흐름이므로 이번 범위 밖이 맞다.

이 다섯 항목은 모두 합류 계약 또는 사용자 확인을 통해 해소 가능하다. 새 게임 기능을 요구하지는 않는다.

## 2. 범위 초과 또는 PRD 위반 가능성

이번 계획은 새 게임 기능, 저장/불러오기, 재시작, 완성형 아트 어느 것도 요구하지 않으며 결정 008(MVP 범위), 결정 009(연대기 스타일), 결정 010(상태 표시 계층화)의 보류 영역을 침범하지 않는다. PRD 위반 위험은 낮다.

다만 다음 두 지점은 결정 위반 직전이므로 합류 계약에서 명시적으로 차단해야 한다.

- **상태 라벨 문구 확정 유혹**: 탭 내부 라벨링 정리 작업 중에 `구간 1/2/3/4` 자리표시자를 "튼튼", "흔들림" 같은 톤이 있는 문구로 바꾸고 싶어질 수 있다. 결정 010의 결정 보류 영역(라벨 문구 — 시대/세계관 톤 결정 라운드 위임)이다. 구현 지시서에 "상태 라벨 문구는 그대로 유지"를 박아두어야 한다.
- **`유산` 탭에서 상태 거동 시각화 시도**: `Reputation 3→2 감쇠`, `Cohesion 4→2 재구성` 같은 차분을 보여주려면 `ApplyPlaceholderStateCarry`가 이전 값을 보존해야 한다. 이는 데이터 구조 변경이며 결정 008의 결정 보류 영역(감쇠율/재구성 강도)에 가깝다. 이번 작업에서는 차분 표시를 시도하지 않는다고 못 박아야 한다.

## 3. 현재 UI 구조 관찰 보완

리뷰 요청서가 빠뜨린 현재 코드 사실 몇 가지를 보완한다.

- **루트 anchors 중복**: `Main._Ready()`가 `SetAnchorsAndOffsetsPreset(FullRect)`을 호출하고, 그 안의 첫 `VBoxContainer root`도 `FullRect`를 또 잡는다(`Main.cs:61-64`). 동작은 하지만 `MarginContainer`/`ScrollContainer`/`TabContainer`를 도입할 때 anchors가 중첩되면서 작은 창 reflow에 미묘한 어긋남이 생긴다. 4영역 구조 재작성 시 루트 `Control`에서만 `FullRect`를 잡고, 내부 컨테이너는 자기 부모 컨테이너에 의해 늘어나도록 통일해야 한다.
- **`Scenes/Main.tscn`은 빈 루트 + 스크립트 어태치뿐**: 새 정적 자식 노드는 없다. `BuildShell()` 재작성으로 모든 영역 변경이 끝나며, `.tscn` 수정 불필요. 케이스 mismatch 함정(CLAUDE.md "ext_resource path 케이스") 회피 차원에서도 `.tscn` 무변경은 옳다.
- **`--smoke`는 UI 노드를 만들지 않는다**: `RunSmoke()`는 `MultigenFlow` public API만 호출하므로 UI shell 재작성은 스모크에 영향 없다. 거꾸로 말하면, **스모크는 이번 UI 변경의 회귀 검증 도구로는 무력하다**. 탭 갱신 누락, 버튼 미생성, ActionBar 갱신 실패는 자동 검증으로 잡히지 않는다. 이 한계는 보고서에 명시돼야 한다.
- **`_pendingEnd` 라이프사이클**: 세대 요약 → AfterSummary 사이 임시 상태로 보관되며, `History.Add`는 `AfterSummary`에서야 일어난다. 즉 세대 요약 화면 시점의 `family.History`에는 **아직 현재 세대가 포함돼 있지 않다**. `가문사` 탭을 세대 요약 화면에서 미리 그릴 때 이 시퀀스를 깨지 말아야 한다(예: 세대 요약 본문에 "이번 세대"를 별도로 표시, 탭은 "지난 세대만").
- **`_family.IsExtinct` 사용처**: 현재 코드에서 `IsExtinct` 플래그는 `MarkExtinct()` 직후 켜지지만 어디서도 읽지 않는다. 멸문 화면 진입 후 ActionBar 진행 버튼을 disabled로 만들고 싶다면 이 플래그를 분기 조건으로 사용할 수 있다.

## 4. 레이아웃 구조 개선 제안

리뷰 요청서의 4영역 구조(`ContextHeader` / `MainScroll(MainContent)` / `InfoTabs` / `ActionBar`)는 Godot Control에서 무리 없이 구성 가능하다. 한 줄 추가 제안:

```text
Control (Main, FullRect)
└─ MarginContainer (작은 화면 안전 여백)
   └─ VBoxContainer AppRoot
      ├─ ContextHeader        SizeFlags: ShrinkBegin
      ├─ ScrollContainer MainScroll
      │  └─ VBoxContainer MainContent  SizeFlags: ExpandFill
      ├─ TabContainer InfoTabs         (CustomMinimumSize.Y 고정 권장)
      └─ HBoxContainer or VBoxContainer ActionBar  SizeFlags: ShrinkEnd
```

핵심 포인트:

- **`MainScroll`만 `ExpandFill`**: 본문이 영역을 차지하고, `ContextHeader`/`InfoTabs`/`ActionBar`는 자기 최소 높이만 차지한다. 그래야 본문이 길어져도 탭과 ActionBar가 화면 밖으로 밀려나지 않는다.
- **`InfoTabs.CustomMinimumSize.Y` 명시**: `TabContainer`는 자식 크기에 따라 늘어나려 한다. 작은 세로 화면에서 탭 안의 `상태` 내용이 길면 ActionBar를 밀어낸다. 탭 자체 내부에도 `ScrollContainer`를 두거나, `InfoTabs`의 최대 높이를 viewport의 30-35% 정도로 고정하는 안을 권장한다.
- **`MarginContainer` 도입**: 작은 화면에서 본문이 화면 가장자리에 닿지 않도록 8-12px 여백을 둔다. 결정 012의 "텍스트 잘림 방지" 검증 기준과 직결.

탭 또는 ActionBar를 사이드(데스크톱 가로 화면)로 옮기는 반응형 분기는 **이번 범위에서 하지 않는다**. 결정 012가 "데스크톱 대형 화면만 전제로 한 넓은 가로 레이아웃을 만들지 않는다"라고 명시했고, 세로 흐름 단일 구조가 단순성·검증 가능성 측면에서 우월하다.

## 5. 탭 구조와 항상 보이는 정보 검토

### 탭 4개 vs 3개

탭 4개(`상태`/`연대기`/`가문사`/`유산`)는 결정 012의 "탭이 너무 많아 주요 흐름을 방해하면 안 된다"의 상한에 가깝지만 위반은 아니다. 다만 **현재 데이터로 `유산` 탭을 의미 있게 채우기 어렵다**.

`유산` 탭의 후보 정보:

- `CarriedOutcomeTags`: 현재 `outcome-1` 자리표시자 문자열 반복. 사용자에게 의미 전달 약함.
- 표시 전용 특성 스텁(`trait-1` / `특성 1`): 어떤 흐름 입력에도 쓰이지 않음. `상태` 탭에 인물 정보로 함께 두는 게 자연스럽다.
- 1세대→2세대 진입 시 상태 거동 차분: 현재 데이터 구조상 보존되지 않음. 보존하려면 범위 초과.

**권장 선택지 A (3개 탭으로 축소)**:

| 탭 | 포함 정보 |
|---|---|
| `상태` | 4단계 라벨, 상세 숫자/방향(sub-section), 현재 인물 특성 |
| `연대기` | 현재 세대 `ChronicleEntry` 목록 |
| `가문사` | `History`의 세대별 요약 + 멸문 시 전체 이력, 누적 `CarriedOutcomeTags` |

**권장 선택지 B (4개 유지, 단 `유산` 콘텐츠 정의 명문화)**:

`유산` 탭은 1세대에서 "[자리표시자: 아직 이어진 유산이 없습니다]"로 두고, 2세대 이상에서만 `CarriedOutcomeTags`와 직전 세대의 `SummaryParagraph` 발췌를 보여준다. 단, 사용자에게 "왜 1세대에서 탭이 비어 보이는가" 혼동을 줄 수 있다.

A안이 단순하고 검증 가능성이 높다. Project Codex가 둘 중 선택하거나 사용자 재확인을 거치도록 권한다.

### 항상 보이는 정보

리뷰 요청서의 `ContextHeader` 5요소(가문/연호/인물/단계/목적 문장)는 적절하다. 한 줄 보완:

- **세대 정보 라벨 단축화**: 현재 헤더가 `가문 A — 가문력 1년 — 인물 1` 형식인데, 작은 폭에서 가로 폭을 차지한다. 두 줄 분할 또는 `ContextHeader`를 `VBoxContainer`로 두고 1행 = 가문/세대/인물, 2행 = 단계/목적 문장으로 분리하는 안을 검토.
- **"현재 단계" 정확한 표현**: 시작/사건/결과/세대 요약/후계/2세대 진입/멸문 7개 단계의 라벨을 합류 계약에서 정해야 화면별로 일관된다. 라벨 자체는 자리표시자(`[단계: 사건 진행 중]` 등)로 두되, 단계 enum/상수를 코드에 도입할 가치가 있다.

## 6. 작은 창/세로형 대응 위험

### `project.godot` 변경의 영향 범위

현재 파일에는 `[display]` 섹션이 없다. Godot 기본값(viewport 1152×648, stretch mode disabled)으로 동작한다. 작은 창과 세로형 검증을 목적으로 한다면 다음 중 최소한이 필요하다:

```ini
[display]

window/size/viewport_width=540
window/size/viewport_height=960
window/stretch/mode="canvas_items"
window/stretch/aspect="keep"
```

영향:

- **다른 실행 환경 영향**: 이 프로젝트는 현재 데스크톱만 타깃이므로 viewport 변경의 부수 효과는 적다. 다만 사용자가 기존에 "기본 윈도우 크기 1152×648"에 맞춘 멘탈 모델을 가지고 있다면 첫 실행이 작아 보일 수 있다. **변경 자체를 사용자에게 재확인**해야 한다.
- **stretch mode 선택**: `canvas_items`는 UI에 권장. `viewport`는 픽셀 아트용. 라벨 텍스트가 흐려지지 않으려면 `canvas_items`.
- **aspect 선택**: `keep`은 검은 띠. `expand`는 콘텐츠 영역 확장. UI 게임이므로 `expand`도 검토 가능. 단 `expand`는 가로/세로 비율 변화 시 빈 공간이 생긴다.

대안: `project.godot`을 건드리지 않고 사용자가 직접 창 크기를 조정해 검증하도록 두는 방법도 있다. 결정 012의 "구체 UI 구현 방식 위임" 원칙상 양쪽 다 허용된다. Project Codex가 사용자에게 선택을 받도록 권한다.

### 사건 화면 선택지 위치

가장 큰 위험. 두 안 비교:

**안 1: 선택지를 MainContent 하단에 둔다**

- 사건 본문과 선택지가 같은 스크롤 안에 있어 의미 단위로 묶인다.
- 본문이 길면 선택지가 화면 밖으로 나가 스크롤해야 누를 수 있다.
- 결정 011의 "다음에 눌러야 할 주요 행동이 명확한가"를 약하게 만든다.

**안 2: 선택지를 ActionBar로 옮긴다**

- 본문 길이와 무관하게 선택지가 항상 보인다.
- 선택지 수가 늘면 ActionBar가 비대해진다(현재는 모든 사건 2개 선택지). MVP 사건이 모두 2개 선택지이므로 당분간은 문제 없음.
- 사건 화면을 제외한 다른 화면의 ActionBar는 단일 진행 버튼이라 ActionBar 의미가 일관되지 않는다(어떤 화면은 1버튼, 어떤 화면은 2버튼+).

**권장**: 안 2를 채택하되, **ActionBar의 자식 수 상한을 합류 계약으로 박는다**(예: 최대 3개, 초과 시 MainContent로 fallback). 현재 MVP 사건은 모두 2개라 무리 없음. 추후 선택지 수가 늘면 합류 계약에서 다시 정한다.

### 탭 라벨 폭

Godot 4 `TabContainer`는 탭 헤더가 가로 배치만 지원하고, 폭 초과 시 좌우 스크롤 버튼을 자동 생성한다. 한국어 2자 라벨 4개는 540px 폭에서도 문제없다. 다만 탭 자식 노드 이름이 곧 탭 헤더 텍스트가 되므로(또는 `set_tab_title(idx, "라벨")`로 별도 지정), 노드 이름과 표시 라벨을 분리하는 패턴을 합류 계약에 둔다.

## 7. 기능 보존 위험과 검증 보완

리뷰 요청서의 위험 표는 적절하다. 추가 위험과 대응:

| 위험 | 대응 |
|---|---|
| 탭 갱신 시점 누락으로 인한 stale 표시 | `RenderInfoTabs(family)` 단일 진입점을 만들고 `ShowGenerationStart`, `ApplyChoice` 직후, `ShowGenerationSummary` 진입, `OnSuccessionPicked` 직후, `ShowExtinction` 진입 시 호출하도록 합류 계약화 |
| `_pendingEnd` 라이프사이클을 깨고 세대 요약에서 `History.Add`를 미리 호출 | `History.Add`는 `AfterSummary` 시점만 유지. 세대 요약에서 `가문사` 탭은 "이번 세대 제외" 상태로 그린다 |
| 멸문 화면에서 ActionBar 진행 버튼이 활성으로 남아 클릭 시 예외 | `_family.IsExtinct` 플래그를 ActionBar 분기 조건으로 사용해 진행 버튼을 만들지 않는다 |
| `BuildShell()` 재작성 중 `_statsPanel`/`_detailPanel`/`_detailToggleButton` 필드가 unused로 남거나 다른 메서드에서 stale 참조 | 4영역 구조 도입과 동시에 옛 필드 전부 제거. 컴파일러가 dangling 참조를 잡는다 |
| 화면 전환 후 `MainScroll` 위치가 직전 화면의 스크롤 위치를 유지해 결과/사건/요약 화면에서 화면이 가운데부터 시작 | 화면 전환 메서드(`ShowEvent`/`ShowResult`/`ShowGenerationSummary` 등) 시작 시 `MainScroll.ScrollVertical = 0` 호출 |
| 탭 전환을 사용자가 사건 진행 중에 한 채로 다음 화면이 와도 자동으로 다시 `상태` 탭으로 돌아가지 않음 | 의도된 동작인지(사용자 선택 유지) 또는 화면 전환 시 항상 0번 탭으로 리셋인지 합류 계약에서 결정. 기본은 "사용자 선택 유지" 권장 |
| 자동 검증(`--smoke`)이 UI 변경의 회귀를 거의 잡지 못함 | 보고서에 명시. 수동 GUI 검증을 자동 검증과 동격으로 다룬다 |
| `BuildShell()`이 비대해져 한 메서드가 100+줄 | 영역별 빌더 메서드 분리(`BuildContextHeader()`, `BuildMainScroll()`, `BuildInfoTabs()`, `BuildActionBar()`)는 허용. 단 새 파일 분리는 하지 않는다 |

### 자동 검증 보완

`--smoke`는 그대로 유지하되, 다음을 보고서에 명시한다:

- 스모크 통과는 `MultigenFlow` 흐름 회귀가 없다는 뜻일 뿐, UI 회귀를 검증하지 않는다.
- UI 회귀는 수동 GUI 검증 체크리스트로만 잡힌다.

선택적 추가(범위 초과 가능성 있음): UI shell이 `null` 노드 없이 빌드되는지만 확인하는 최소 헤드리스 UI 스모크(`--ui-smoke`). 의의는 작고, 사람 검증으로 충분하므로 **이번 작업에서는 도입하지 않음을 권장**한다.

## 8. 구현 전 반드시 정해야 할 합류 계약

Project Codex가 최종 작업 지시서에 명시해야 할 합류 계약 후보:

### UI 구조 계약

- `Scenes/Main.tscn`은 무변경. 루트 `Control` + `res://scripts/Main.cs` 어태치만 유지. 새 정적 노드를 만들지 않는다.
- `Main.cs`는 새 파일로 분리하지 않는다. UI shell은 `BuildContextHeader`/`BuildMainScroll`/`BuildInfoTabs`/`BuildActionBar` 영역별 빌더 메서드로 나누되 같은 파일에 둔다.
- 4영역 외 영역을 새로 만들지 않는다.

### 단일 갱신 지점

- 탭 갱신: `RenderInfoTabs(FamilyRunState family)` 한 곳에서 모든 탭을 다시 그린다.
- 호출 지점: `ShowGenerationStart` 진입, `ApplyChoice` 직후, `ShowGenerationSummary` 진입, `OnSuccessionPicked` 직후, `ShowExtinction` 진입.
- ActionBar 갱신: 각 화면 메서드가 `SetActionBar(...)` 단일 호출로 갱신.
- ContextHeader 갱신: 화면 전환마다 `RefreshContextHeader()` 호출.

### 멸문 화면 종료 처리

- 멸문 화면 ActionBar에는 진행 버튼을 두지 않는다.
- 대신 비활성/안내 라벨(예: `[자리표시자: 가문이 멸문하여 더 이상 진행할 수 없습니다]`)을 ActionBar 영역에 표시한다.
- 재시작 버튼은 추가하지 않는다.

### 상세 정보 처리

다음 두 안 중 하나를 사용자 확인 또는 Project Codex 판단으로 확정:

- **안 A**: 상세 토글 유지. `상태` 탭은 4단계 라벨만, 토글은 `상태` 탭 안 sub-section으로 둔다.
- **안 B**: 토글 제거. `상태` 탭에 4단계 라벨 + 상세 숫자/방향을 sub-section으로 함께 표시.

결정 010의 계층화 원칙 충족을 위해 어느 쪽이든 "기본 라벨 영역"과 "상세 영역"이 시각적으로 분리돼야 한다.

### `유산` 탭 처리

다음 두 안 중 하나를 사용자 확인 또는 Project Codex 판단으로 확정:

- **안 A (권장)**: 탭 3개(`상태`/`연대기`/`가문사`)로 축소. 유산 정보는 `가문사` 탭 상단에 누적 `CarriedOutcomeTags` 줄을 둔다.
- **안 B**: 탭 4개 유지. 1세대에서는 `유산` 탭에 비어 있음 안내 자리표시자 표시.

### `project.godot` 변경 범위

사용자 확인 필요. 변경한다면 `[display]` 섹션 추가만 허용하고 viewport 540×960, stretch `canvas_items`, aspect 사용자 선택. 변경하지 않는다면 사용자가 창 크기를 직접 조정해 검증.

### 콘텐츠 무변경

- 가문명/인물명/도입 문장/요약/사건 문안은 변경하지 않는다.
- 상태 4단계 라벨(`구간 1`~`구간 4`)도 변경하지 않는다.
- "현재 단계" 표시 문구만 새로 추가 가능하되 모두 `[단계: ...]` 자리표시자 형식.

### `--smoke` 무변경

- `--smoke` 코드 경로는 UI 노드를 만들지 않는다는 합류 계약을 유지한다.
- `--smoke` stdout 출력 라인도 변경하지 않는다.

## 9. 수동 UI/UX 검증 기준 보완

리뷰 요청서의 검증 항목과 `docs/manual-ui-ux-validation-guide.md`는 잘 맞물려 있다. 다음을 추가 또는 보완한다.

### 정량 기준 추가

- **검증 창 크기 3종 명시**: 360×640(세로형 모바일 가상 화면), 540×720(좁은 데스크톱 창), 1280×720(일반 데스크톱). 사용자가 자기 환경 1종만 통과시키고 완료 판단하는 것을 막는다.
- **검증 시퀀스 1회 분량 측정**: 시작부터 멸문까지 1회 ~30-60초로 짧으므로, 3종 창 크기에서 1회씩 총 3회 플레이로 검증한다.

### 동작 검증 추가

- **탭 갱신 시점**: 선택 적용 직후 `상태`/`연대기` 탭이 실제로 업데이트되는가. 세대 전환 직후 `가문사` 탭에 직전 세대가 추가되는가. 멸문 직후 `가문사` 탭에 전체 이력이 보이는가.
- **탭 사용자 선택 유지**: `연대기` 탭을 연 채로 다음 사건/결과로 진행해도 사용자의 탭 선택이 유지되는가(또는 의도적으로 `상태` 탭으로 리셋되는가). 합류 계약과 동작이 일치하는지 확인.
- **스크롤 리셋**: 화면 전환 시 본문 스크롤이 최상단으로 돌아오는가.
- **멸문 화면의 종료성**: ActionBar에 진행 버튼이 없는가. 클릭 가능한 요소가 탭 헤더뿐인가.

### 자리표시자 가시성 확인

- 자리표시자 문구(`[자리표시자: ...]`, `[단계: ...]`, `구간 1`~`구간 4`)가 시각적으로 자리표시자임을 알 수 있는가.
- 자리표시자가 최종 제품 문안처럼 오해될 위험이 있는 화면이 있는가.

### 자동 검증과 분리 보고

- `dotnet build` / `--import` / `--smoke` 3개 명령 결과는 "기능 흐름 회귀 없음"만 의미한다.
- UI 구조 회귀는 수동 GUI 체크리스트로만 보고한다.
- 두 결과를 같은 보고서에서 별도 섹션으로 분리한다.

## 10. Project Codex가 사용자에게 재확인해야 할 항목

다음은 Project Codex가 최종 작업 지시서를 확정하기 전 사용자에게 직접 묻는 것을 권한다. Project Claude가 추측으로 결정할 사안이 아니다.

1. **`project.godot`의 `[display]` 섹션 추가 허용 여부 + viewport 기준 크기**
   - 추가하지 않고 사용자가 창 크기를 직접 조정해 검증할지, 추가한다면 540×960 등 어떤 기준 크기를 사용할지.

2. **`유산` 탭 처리 (안 A 3개 탭 vs 안 B 4개 탭)**
   - 자리표시자 데이터만 있는 `유산` 탭을 별도로 두는 것이 UX 측면에서 가치 있는지, 아니면 `가문사`에 흡수하는 게 단순한지.

3. **상세 정보 처리 (안 A 토글 유지 vs 안 B 토글 제거)**
   - 결정 010의 계층화 원칙을 충족시키는 두 안 중 어느 쪽을 선호하는지.

4. **멸문 화면 종료 표시 수준**
   - "[자리표시자: 가문이 멸문하여 더 이상 진행할 수 없습니다]" 같은 비활성 안내 라벨만 둘지, 아니면 별도 처리 원하는지.

5. **탭 사용자 선택 유지 여부**
   - 화면 전환 시 사용자가 마지막으로 열어둔 탭을 유지할지, 항상 `상태` 탭으로 리셋할지.

6. **검증 창 크기 3종 (360/540/1280) 적용 동의**
   - 사용자 검증 환경에서 3종 창 크기를 모두 시도할 수 있는지, 또는 다른 기준을 원하는지.

이 6개 항목 중 1, 4, 6은 사용자 환경/선호 의존이라 Project Codex/Claude가 단독 결정하기 부적절하다. 2, 3, 5는 Project Codex가 결정해도 무방하나 사용자 확인을 받으면 후속 수정 요청을 줄일 수 있다.
