## 목적

현재 다세대 MVP는 기능 경로가 구현되어 있다. 이번 작업의 목적은 기능을 새로 늘리는 것이 아니라, 사용자가 작은 창과 세로형 모바일 화면에서도 시작부터 멸문까지 직접 플레이하며 UI/UX를 검증할 수 있도록 정보 구조와 조작 흐름을 정리하는 것이다.

제품 목적은 다음이다.

- 화면마다 목적과 다음 행동을 즉시 이해할 수 있게 한다.
- 사건, 선택지, 결과, 상태 변화, 연대기 기록이 서로 섞여 보이지 않게 한다.
- 현재 가문, 세대, 인물, 현재 단계는 항상 확인 가능하게 한다.
- 상태, 연대기, 이전 세대 기록, 큰 사건 결과, 특성 스텁은 탭 또는 보조 영역에서 확인하게 한다.
- 기능 검증과 UI/UX 수동 검증을 분리해서 보고할 수 있게 한다.

이번 문서는 리뷰 요청서이며 실제 구현은 하지 않는다.

## 현재 구조 관찰

### 구현 상태

`multigen-mvp` handoff 기준으로 다세대 MVP 기능은 완료 보고됐다.

- `dotnet build`: 성공 보고
- Godot headless import: 성공 보고
- `--smoke`: 1세대 후계 정상 경로와 2세대 멸문 경로 성공 보고
- 사용자 수동 GUI 확인: 완료 보고

### 현재 UI 구현

현재 UI는 `scripts/Main.cs`가 코드 기반 동적 UI를 생성한다.

- `Scenes/Main.tscn`은 루트 `Control` 하나와 `res://scripts/Main.cs` 어태치만 가진다.
- `BuildShell()`은 루트 `VBoxContainer`에 다음을 순서대로 배치한다.
  - 헤더 라벨
  - `_content` 본문 영역
  - 구분선
  - 상태 라벨 영역
  - 상세 정보 보기/숨기기 버튼
  - 상세 정보 패널
- 시작, 사건, 결과, 세대 요약, 후계, 멸문 화면은 모두 `_content`를 비우고 다시 채우는 방식이다.
- 상태와 특성은 본문 아래 고정 영역에 표시된다.
- 상세 정보는 토글로만 보이며, 연대기/이전 세대 기록/큰 사건 결과/특성 스텁은 별도 탭 구조로 분리되어 있지 않다.

## 현재 UI의 예상 문제점

- 스크롤 컨테이너가 없어 작은 창에서 긴 본문, 선택지, 연대기, 세대 이력이 잘리거나 접근하기 어려울 수 있다.
- 현재 맥락 라벨은 가문/연호/인물만 표시하고, 현재 화면 목적이나 다음 주요 행동이 충분히 분리되어 보이지 않는다.
- 상태, 특성, 상세 정보가 본문 아래에 붙어 있어 긴 본문과 함께 과밀해질 수 있다.
- 사건 본문, 선택지, 결과 문구, 상태 변화, 연대기 기록이 모두 같은 세로 흐름에 쌓여 정보 역할이 즉시 구분되지 않을 수 있다.
- 세대 요약 화면은 누적 연대기, 후대 요약, 이전 세대 이력, 결과 태그가 한 화면에 모두 들어가 작은 폭에서 부담이 클 수 있다.
- 후계 화면과 멸문 화면의 다음 행동 또는 종료 상태가 화면 구조상 강하게 드러나지 않을 수 있다.
- 상세 정보는 상태 숫자 중심이며, 연대기/가문사/결과 태그/특성 스텁을 필요할 때 탭으로 확인하는 구조가 없다.
- UI 구조가 `Main.cs` 단일 파일의 화면별 메서드에 직접 누적되어 있어, 이후 화면 추가/삭제 시 정보 영역 경계가 더 흐려질 수 있다.

## UI/UX 개선 계획

### 기본 방향

기존 기능 흐름과 `--smoke` 경로는 유지한다. 이번 개선은 정보 구조와 조작 흐름 정리에 집중한다.

구현 방식은 현재 구조에 맞춰 코드 기반 동적 UI 생성을 계속 사용한다. `Scenes/Main.tscn`은 루트 `Control` + `res://scripts/Main.cs` 구조를 유지한다. 새 정적 씬 노드 트리를 만들지 않는다.

단, `Main.cs` 내부의 UI 구조를 다음 4개 영역으로 명확히 나눈다.

- 고정 맥락 헤더
- 스크롤 가능한 주요 내용 영역
- 탭 기반 보조 정보 영역
- 고정 주요 행동 영역

### 제안 레이아웃 구조

루트 구조 후보:

```text
Control
└─ MarginContainer
   └─ VBoxContainer AppRoot
      ├─ ContextHeader          (항상 보임)
      ├─ ScrollContainer MainScroll
      │  └─ VBoxContainer MainContent
      ├─ TabContainer InfoTabs  (보조 정보)
      └─ VBoxContainer ActionBar (항상 보임 또는 화면 하단 고정)
```

설계 의도:

- `ContextHeader`: 가문, 세대, 인물, 현재 단계, 화면 목적을 항상 보여준다.
- `MainScroll`: 사건 본문, 결과 문구, 세대 요약 본문처럼 긴 텍스트를 스크롤로 감싼다.
- `InfoTabs`: 상태, 연대기, 가문사, 특성/유산 같은 보조 정보를 탭으로 분리한다.
- `ActionBar`: 시작, 선택, 다음 사건, 세대 요약, 후계 선택, 멸문 확인 같은 주요 행동 버튼을 분명하게 보여준다.

Project Claude 리뷰에서 확인받을 점:

- Godot `TabContainer`가 작은 폭에서 탭 라벨을 다루기에 충분한지
- `ActionBar`를 고정 하단에 두는 것이 현재 동적 UI와 충돌하지 않는지
- 선택지처럼 버튼이 여러 개인 화면에서 `ActionBar`가 과도하게 커지지 않는지

### 항상 보이는 정보

항상 보이는 정보는 `ContextHeader`와 `ActionBar`에 둔다.

`ContextHeader`:

- 가문명
- 현재 연호 또는 세대: 예 `가문력 1년`
- 현재 인물
- 현재 단계: 시작, 사건, 결과, 세대 요약, 후계, 다음 세대 진입, 멸문
- 한 줄 목적 문장: 예 `[자리표시자: 사건을 읽고 선택하세요]`

`ActionBar`:

- 현재 화면의 주요 행동 버튼
- 보조 버튼이 필요하면 최대 1개까지 제한
- 사건 화면에서는 선택지 버튼을 세로 목록으로 보여준다.
- 탭을 보고 있는 중에도 진행 행동이 어디 있는지 알 수 있게 한다.

### 탭 또는 보조 영역으로 분리할 정보

탭 수는 4개를 기본으로 제안한다.

| 탭 | 역할 | 포함 정보 |
|---|---|---|
| `상태` | 현재 판단에 필요한 상태 요약 | 4단계 질적 라벨, 상세 숫자/방향, 특성 스텁 |
| `연대기` | 현재 세대의 선택 기록 | 이번 세대 `ChronicleEntry`, 선택 결과 기록 |
| `가문사` | 이전 세대 기록 | `FamilyRunState.History`, 세대별 요약, 멸문 시 전체 이력 |
| `유산` | 세대 간 이어짐 정보 | `CarriedOutcomeTags`, 이전 세대에서 이어진 상태/큰 사건 결과, 자리표시자 유산 |

탭 명칭은 최종 UI 문구가 아니다. Project Claude는 구현 편의상 더 짧은 라벨을 제안할 수 있다. 단, 탭 수가 5개 이상으로 늘어나면 작은 폭에서 과밀할 가능성이 커지므로 리뷰에서 위험으로 보고한다.

### 화면별 목적과 주요 행동

| 화면 | 목적 | 주요 행동 | 보조 정보 |
|---|---|---|---|
| 시작 | 새 가문과 1세대 진입 맥락 이해 | `첫 사건으로` | 상태, 특성 |
| 사건 | 사건 본문을 읽고 선택 | 선택지 2개 이상 | 상태, 현재 세대 연대기 |
| 결과 | 선택 결과, 상태 변화, 연대기 기록 확인 | `다음 사건으로` 또는 `세대 요약 보기` | 상세 상태, 연대기 |
| 세대 요약 | 한 세대 종료와 유산 이해 | `후계를 본다` 또는 `가문의 끝을 본다` | 가문사, 유산, 상태 |
| 후계 | 다음 세대로 이어지는 이유와 후보 확인 | 후보 선택 또는 `다음 세대로` | 특성, 유산 |
| 2세대 진입 | 이전 세대 흔적이 이어졌음을 확인 | `첫 사건으로` | 가문사, 유산, 상태 |
| 멸문 | 가문 종료와 전체 이력 확인 | `종료 확인` 또는 `다시 보기` 수준의 비기능 버튼은 신중히 검토 | 가문사, 연대기, 유산 |

주의:

- 저장/불러오기, 다시 시작 기능은 이번 제외 범위다.
- 멸문 화면에 실제 재시작 버튼을 넣으면 새 흐름이 될 수 있으므로, 이번 계획에서는 "종료 상태가 명확한 화면"까지만 요구한다.

### 작은 창/세로형 모바일 대응 전략

- 전체 UI는 세로 흐름을 기본으로 한다.
- 본문은 반드시 `ScrollContainer` 안에 둔다.
- 긴 텍스트 라벨은 `AutowrapMode`를 사용한다.
- 버튼은 세로 목록으로 쌓고, 버튼 텍스트가 잘리지 않도록 최소 높이와 줄바꿈 가능성을 검토한다.
- 한 화면에 모든 정보를 노출하지 않고, 상태/연대기/가문사/유산은 탭으로 분리한다.
- 기본 플레이 흐름에서는 `MainContent`와 `ActionBar`만 읽어도 진행 가능해야 한다.
- 보조 탭은 필요할 때 확인하는 정보로 두며, 주요 진행 버튼을 탭 내부 깊숙이 숨기지 않는다.
- 작은 창 수동 검증 기준은 "가로가 좁은 상태에서도 현재 맥락, 본문, 주요 버튼을 찾을 수 있는가"로 둔다.

### 구현 예상 변경 파일

수정 후보:

- `scripts/Main.cs`
  - `BuildShell()` 구조 재작성
  - 화면별 렌더링을 `MainContent`, `InfoTabs`, `ActionBar`, `ContextHeader`로 분리
  - `RenderStats`, `RenderDetailPanel`, `RenderTraits`를 탭 렌더링으로 이동 또는 재구성
  - 화면 단계 enum 또는 문자열 상태 도입 검토
  - 수동 검증용 화면 목적 문구 추가
- `Scenes/Main.tscn`
  - 기존 루트 `Control` + `res://scripts/Main.cs` 유지
  - 변경하지 않는 것이 원칙
- `project.godot`
  - 기본 창 크기나 stretch 설정 변경이 필요한지 리뷰 요청
  - 변경 시 작은 창/세로형 검증 목적에 한정
- `docs/handoff/multigen-ui-ux-improvement/04-implementation-report.md`
  - 구현 결과와 수동 검증 기준 보고

새 파일은 기본 계획에서는 만들지 않는다.

이유:

- 현재 UI가 `Main.cs` 동적 생성 구조로 일관되어 있다.
- 이번 작업은 완성형 UI 시스템이 아니라 구조 정리와 수동 검증 가능성 확보가 목적이다.
- 새 UI 빌더 추상화는 Project Claude 리뷰에서 `Main.cs`가 과도하게 커질 위험이 있다고 판단될 때만 제안받는다.

### 기능 보존 위험과 대응

| 위험 | 대응 |
|---|---|
| 화면 구조 변경 중 `--smoke` 경로가 깨짐 | 기존 `RunSmoke()`는 기능 검증 기준으로 유지하고 변경 후 반드시 실행 |
| 선택지 버튼을 ActionBar로 옮기며 이벤트 캡처가 깨짐 | 버튼 생성과 `ApplyChoice` 연결을 기존 패턴으로 유지 |
| 탭 갱신이 상태 변경 후 누락됨 | 선택 적용, 세대 전환, 멸문 처리 후 `RenderInfoTabs()` 같은 단일 갱신 지점 호출 |
| 세대 요약/멸문 화면에서 정보가 탭으로 숨어 핵심 흐름이 약해짐 | 핵심 요약은 MainContent에 유지하고, 전체 이력/상세는 탭에 둠 |
| 작은 창에서 ActionBar가 본문을 밀어냄 | MainContent만 스크롤되게 하고 ActionBar 버튼 수를 최소화 |
| 탭 라벨이 좁은 폭에서 과밀 | 탭 수 4개 이하 유지, 짧은 라벨 사용 |
| `Scenes/Main.tscn` 경로 케이스 오류 | 기존 `res://scripts/Main.cs` 참조 유지 |

## 구현 단계 초안

1. 현재 기능 흐름 보존 기준을 확인한다.
   - `RunSmoke()` 경로와 주요 화면 메서드를 파악한다.
   - 변경 전후 기능 경로를 비교할 기준을 정한다.
2. `Main.cs`의 UI shell을 4영역으로 재구성한다.
   - `ContextHeader`
   - `MainScroll`/`MainContent`
   - `InfoTabs`
   - `ActionBar`
3. 각 화면 렌더링 메서드를 주요 내용과 행동 버튼 렌더링으로 분리한다.
   - 시작/사건/결과/세대 요약/후계/멸문
4. 보조 정보 탭을 구현한다.
   - 상태
   - 연대기
   - 가문사
   - 유산
5. 작은 창/세로형 대응을 적용한다.
   - 본문 스크롤
   - 라벨 줄바꿈
   - 버튼 세로 배치
   - 탭 수와 라벨 길이 제한
6. 자동 기능 검증을 실행한다.
   - `dotnet build`
   - `.\scripts\godot.cmd --headless --import` 또는 동등 래퍼
   - `.\scripts\godot.cmd --headless -- --smoke` 또는 동등 래퍼
7. 수동 UI/UX 검증 기준으로 사람이 직접 확인할 항목을 보고서에 남긴다.

## 검증 기준

### 기능 검증

Project Claude는 구현 후 다음을 실행하고 보고한다.

```powershell
dotnet build
.\scripts\godot.cmd --headless --import
.\scripts\godot.cmd --headless -- --smoke
```

셸 환경상 `godot.sh`를 써야 하는 경우 동등 래퍼를 사용하고 이유를 보고한다.

### UI/UX 수동 검증 기준

기능 검증과 별도로 다음을 확인한다.

- 시작부터 멸문까지 직접 플레이할 수 있다.
- 작은 창 또는 세로형 창에서 본문과 버튼이 잘리거나 겹치지 않는다.
- 항상 가문, 세대, 인물, 현재 단계가 보인다.
- 각 화면에서 다음 주요 행동이 분명하다.
- 사건 본문과 선택지가 구분된다.
- 결과 문구, 상태 변화, 연대기 기록이 구분된다.
- 세대 요약에서 유산과 다음 세대 상황 암시가 읽힌다.
- 후계 화면에서 다음 세대로 넘어가는 이유와 조작이 명확하다.
- 2세대 진입 시 이전 세대의 상태/연대기/결과 태그가 이어졌음을 탭에서 확인할 수 있다.
- 멸문 화면에서 가문 종료와 전체 이력을 확인할 수 있다.
- 상태/연대기/가문사/유산 탭이 기본 플레이 흐름을 방해하지 않는다.
- 자리표시자 문구가 최종 문안처럼 오해되지 않는다.

## Project Claude 리뷰 요청 사항

Project Claude는 다음을 우선 검토한다.

1. `Main.cs` 안에서 UI shell을 재구성하는 방식이 현재 규모에 적절한지
2. 새 UI 빌더 파일 없이 진행해도 유지보수 가능한지
3. `ContextHeader` + `MainScroll` + `InfoTabs` + `ActionBar` 구조가 Godot Control에서 무리 없는지
4. `TabContainer`가 작은 폭/세로형 검증 기준에 적절한지
5. 탭 4개(`상태`, `연대기`, `가문사`, `유산`)가 과하지 않은지
6. 주요 행동 버튼을 `ActionBar`로 분리할 때 선택지 2개 이상 화면이 과밀하지 않은지
7. 현재 `--smoke`를 그대로 기능 검증 기준으로 유지할 수 있는지
8. `project.godot`의 window/stretch 설정을 바꿔야 하는지, 바꾼다면 범위가 UI/UX 검증 목적에 한정되는지
9. 수동 UI/UX 검증 체크리스트가 구현 결과 보고에 충분한지
10. 이 계획이 완성형 UI/아트 스타일 확정으로 오해될 위험이 있는지

## 멈춤 지점

다음 판단이 필요하면 Project Claude는 리뷰 의견으로 돌려준다.

- UI 개선을 위해 PRD에 없는 새 게임 기능을 추가해야 하는 경우
- 저장/불러오기, 재시작, 설정 화면 같은 새 흐름이 필요해지는 경우
- 완성형 색상, 폰트, 장식 스타일 결정을 요구하는 경우
- 상태 이름, 라벨 문구, 밸런스 수치 확정을 요구하는 경우
- 탭 구조가 현재 기능 흐름을 깨뜨릴 위험이 큰 경우
- `Main.cs` 변경이 너무 커져 새 UI 파일 분리가 필요하다고 판단되는 경우
- 작은 창 대응을 위해 Godot 프로젝트 설정을 바꾸는 것이 다른 실행 환경에 영향을 줄 수 있는 경우

## 사용자 수동 검증 안내 초안

구현 후 사용자는 다음 순서로 플레이한다.

1. 실행 후 시작 화면을 본다.
2. 헤더에서 가문, 세대, 인물, 현재 단계가 보이는지 확인한다.
3. 첫 사건으로 들어간다.
4. 사건 본문과 선택지 버튼이 구분되는지 확인한다.
5. `상태`, `연대기`, `가문사`, `유산` 탭을 한 번씩 열어본다.
6. 선택지를 누르고 결과 화면으로 간다.
7. 결과 문구, 상태 변화, 연대기 기록이 서로 분리되어 보이는지 확인한다.
8. 다음 사건으로 진행해 1세대 사건 3개를 끝낸다.
9. 세대 요약에서 유산, 다음 세대 암시, 상태 서술이 보이는지 확인한다.
10. 후계 화면에서 다음 세대로 넘어가는 행동이 명확한지 확인한다.
11. 2세대 진입 후 이전 세대 이력과 결과 태그가 탭에서 보이는지 확인한다.
12. 2세대 사건 3개를 진행하고 멸문 화면까지 간다.
13. 멸문 화면에서 가문 종료와 전체 세대 이력을 확인한다.
14. 창을 좁거나 세로형으로 조정해 텍스트와 버튼이 잘리거나 겹치지 않는지 확인한다.

## Project Codex 보고 요약

### 계획 요약

현재 `Main.cs`의 단순 세로 누적 UI를 유지보수 가능한 4영역 구조로 재배치한다. 헤더는 항상 가문/세대/인물/현재 단계를 보여주고, 본문은 스크롤 가능한 주요 내용으로 분리한다. 상태, 연대기, 가문사, 유산은 탭으로 이동하며, 주요 행동 버튼은 화면 하단의 행동 영역에서 명확히 제공한다.

- 주요 행동 버튼

- `상태`
- `연대기`
- `가문사`

### Project Claude에게 확인받을 질문
