## 주요 검토 의견

### 1. 아키텍처 규모는 적절. 단일 책임 경계 한 곳만 손볼 만함

- `Scripts/MvpLoopModels.cs`, `Scenes/Main.tscn`, `Scripts/Main.cs`, `project.godot` 수정 — 이 4점 변경 자체는 현재 스켈레톤(`.cs`/`.tscn` 0개) 대비 과하지 않다. 결정 003–006이 요구하는 9단계 사용자 흐름을 정확히 덮는다.
- 다만 `MvpLoopModels.cs` 한 파일이 **모델 타입 정의 + 3개 사건의 시드 데이터**를 모두 짊어진다. 결정 006이 "사건 데이터 저장 형식과 구현 구조"를 의도적으로 미확정 영역으로 명시했는데, 모델과 시드 데이터가 한 파일/한 네임스페이스에 묶이면 이후 단계에서 이 형태가 "확정된 저장 형식"으로 굳어질 위험이 있다. **타입 정의(`MvpLoopModels.cs`)와 시드 콘텐츠(`MvpLoopContent.cs`)를 파일 단위로 분리** 권장. 같은 폴더, 같은 어셈블리, 추가 비용 거의 0. 그 대신 "콘텐츠는 검증용 임시 시드"라는 의도가 파일 이름 한 줄로 드러난다.

### 2. 합류 계약 후보 — 선택지 2(코드 기반 UI 동적 생성) 지지

Project Codex의 선호와 같다. 근거 보강:

- 결정 006이 "세대 요약 화면의 최종 UI 형식"과 "완성형 UI/아트 스타일"을 의도적 미확정으로 못 박았다. 씬에 노드를 미리 박는 비용 대비 회수 가치가 없다.
- 작업 B↔C 합류 계약이 "노드 경로 N개"에서 "루트 Control 노드 1개 + 스크립트 어태치" 한 줄로 줄어든다. 충돌 표면이 크게 축소된다.
- C# 코드 안에서 `VBoxContainer`/`Button`/`Label`을 즉시 생성·바인딩하는 패턴은 Godot에서 표준. MVP 검증 후 정식 씬으로 전환하는 비용도 작다.

따라서 선택지 2 채택. 단, "Main.cs는 `_Ready()` 시점에 모든 UI를 동적으로 만든다"를 합류 계약 항목(아래 4번)에 명시해야 한다.

### 3. 상태 항목 3개(`Reputation`/`Stores`/`Cohesion`)는 PRD 안. 단 키와 표시명을 분리할 것

- 결정 006은 "선택 결과는 적어도 **하나의** 상태 항목 변화"만 요구. 3개는 상한이 아닌 자연수다. 다만 항목이 늘수록 사건마다 어느 키를 건드릴지 결정해야 하고, 그 결정이 콘텐츠 부담으로 돌아온다.
- 결정 004·006이 한국어 표시 흐름을 가정한다(가문/인물명, 도입 문장 모두 한국어). 영문 키 `Reputation`은 식별자, 표시명은 한국어("명망" 등)로 분리해야 PRD 표시 흐름과 맞는다. 제안 문서의 `상태 키, 표시명, 변화량` 분해는 이미 이 방향이라 OK. 다만 시드 단계에서 표시명까지 임시값으로 채우고 "콘텐츠 미확정"임을 코드 주석에 한 줄 남길 것.

### 4. `LoopEvent`/`EventChoice`/`StateDelta`/`ChronicleEntry` 모델 분해는 적절

모델 4종이 각자 단일 책임을 가진다(사건, 선택, 상태 변화, 연대기 기록). 결정 005·006이 요구하는 "서사 결과 + 연대기 항목 + 최소 상태 변화"의 세 갈래를 그대로 1:1 매핑한다. 더 줄여 익명 튜플로 가는 것은 권하지 않는다 — `Main.cs`의 분기 가독성이 떨어진다.

## 병렬 작업 분해 개선 제안

### 1. 합류 계약 2(코드 UI 동적 생성)를 채택하면 작업 B는 사실상 비어짐

작업 B의 산출물이 "빈 Control 루트 + `Main.cs` 어태치 + `project.godot`의 `run/main_scene` 한 줄"로 축소된다. 단독 트랙으로 운영할 가치가 거의 없다.

**권장: 작업 B를 작업 C에 흡수, 2트랙(A: 모델·콘텐츠, C: 씬+로직)으로 단순화.** `Main.cs`를 작성하는 사람이 같은 호흡에 `Main.tscn`을 만들고 `project.godot`을 한 줄 고친다. 노드 경로 합류 계약이 사라지므로 충돌 위험이 0에 수렴한다.

### 2. 작업 A를 둘로 더 쪼개지 말 것

모델과 시드를 파일 단위로 분리(주요 검토 의견 1번)하더라도 한 작업자가 두 파일을 같이 쓰는 게 자연스럽다. 모델 시그니처가 시드 데이터의 형태와 합의되어야 하기 때문. 둘을 다른 트랙으로 쪼개면 합류 계약 수가 늘어 손해다.

### 3. 작업 A↔C 병렬화의 진짜 조건

작업 A가 **public 타입의 최종 시그니처**와 **시드 데이터 접근 진입점**(예: `MvpLoopContent.Events`, `MvpLoopContent.CreateInitialRunState()`)을 먼저 한 번에 확정해야 작업 C가 컴파일 가능한 스텁으로 진행할 수 있다.

합의 전에 두 작업을 동시에 시작하면 작업 C가 가짜 stub로 진행되다가 재작업이 발생한다. **선행 의존성: "구현 전 반드시 정해야 할 합류 계약 1–5 문서화 → 그 후 병렬 진행"**.

## 구현 전 반드시 정해야 할 합류 계약

작업 A와 작업 C 사이:

### 1. 모델 타입 시그니처

- `RunState` 프로퍼티: `FamilyName`, `FounderName`, `IntroLine`, `CurrentEventIndex`, `Stats`(키→정수 누적값), `Chronicle`(기록 항목 누적).
- `LoopEvent.Choices` 컬렉션 타입: `IReadOnlyList<EventChoice>` 권장(불변 시드, 인덱스 접근).
- `StateDelta`: `Key`(string) + `Amount`(int). 표시명은 `StateDefinition` 측에서 조회(5번 항목 참고). `StateDelta`가 표시명을 함께 들고 다닐지 한 군데에서 조회할지 한쪽으로 통일.
- `ChronicleEntry`: `EventTitle` + `ChoiceLabel` + `RecordedLine`. 표시 순서는 입력 순서 그대로.

### 2. 시드 데이터 접근 진입점

정적 진입점 1곳으로 한정.

- 권장: `MvpLoopContent.Events`(읽기 전용 컬렉션) + `MvpLoopContent.CreateInitialRunState()`.
- `Main.cs`는 위 두 곳만 호출. 그 외 시드 접근 금지.

### 3. 결과 적용 책임 위치

권장: `RunState.ApplyChoice(EventChoice)`를 모델 측에 두고 `Stats` 누적과 `Chronicle` 추가를 한 메서드에서 수행. `Main.cs`는 호출만. 콘텐츠↔흐름 책임이 깔끔히 갈라지고, 누락(예: 연대기는 추가했는데 상태 누적 누락) 가능성이 낮아진다.

### 4. UI 합의 — 동적 생성 채택 시 파일 책임

- `Scenes/Main.tscn`: 루트 `Control`만 + `Main.cs` 어태치. 자식 노드 없음.
- `project.godot`: `[application] run/main_scene="res://Scenes/Main.tscn"` 한 줄.
- `Main.cs`: `_Ready()`에서 컨테이너·라벨·버튼을 모두 생성하고 상태 전환(시작/사건/결과/세대 요약)을 단일 메서드 디스패치로 처리.

### 5. 상태 항목 정의 보관 위치

- 후보 A: `Stats`를 `Dictionary<string,int>`로 두고 표시명은 시드에서 매번 같이 흘려보내기.
- 후보 B: `StateDefinition(string Key, string Label)` 레코드 배열을 `MvpLoopContent`에 한 번만 두고 어디서나 참조.
- **권장: 후보 B.** 표시명 일관성, 키 오타 방지.

## 검증 방법 보완 제안

### 현재 명령의 한계

- `dotnet build` → C# 컴파일만 검증.
- `.\scripts\godot.cmd --headless --import` → 에셋 임포트와 `.godot/` 갱신만 검증. **사용자 흐름 9단계는 단 한 칸도 검증되지 않는다.**
- `.\scripts\godot.cmd --headless --quit-after 2` → Godot 4의 `--quit-after`는 **프레임 단위**다. 2프레임 안에 사건 진행/선택/세대 요약이 발생할 수 없다. **현재 형태로는 사용자 흐름을 검증하지 못한다.**

### 보완안 — 두 단계 검증

사용자 흐름이 9단계로 정의된 이상, 컴파일/임포트만으로 "구현 완료"를 주장할 수 없다.

**(1) 자동 스모크(헤드리스)**

- `Main.cs`에 `--smoke` 인자(또는 `LEGACY_LOG_SMOKE=1` 환경 변수) 분기를 추가. 분기 진입 시:
  1. 새 실행 초기화 → 시작 정보 출력(stdout).
  2. 각 사건에서 첫 번째 선택지를 자동 선택.
  3. 선택 직후 결과 문구·상태 변화·연대기 항목을 stdout으로 1행씩 기록.
  4. 세대 요약까지 도달하면 `SMOKE_OK`를 출력하고 `GetTree().Quit()`.
- 실행: `.\scripts\godot.cmd --headless -- --smoke` (또는 환경 변수 + 일반 헤드리스).
- 종료 코드 0 + stdout에 `SMOKE_OK`가 있으면 자동 통과. 종료 코드 0이라도 `SMOKE_OK` 없으면 실패로 판정.
- 이 분기는 **MVP 검증용 임시 코드**이므로 추후 통합 테스트 도입 시 제거 대상임을 코드 주석에 명시.

**(2) 수동 GUI 1회**

- 사람이 `.\scripts\godot.cmd -e`(에디터)나 `.\scripts\godot.cmd`(런타임)로 한 번 실행해 9단계가 화면에서 모두 보이는지 확인.
- 결정 003·005가 요구하는 "기록처럼 읽히는 출력"은 stdout만으로 검증 불가. **수동 1회 확인을 보고 형식의 필수 항목으로 추가** 권장.

### 보고 항목 보강

`docs/current-focus.md`의 Project Claude 보고 형식("변경 파일 / 구현된 사용자 흐름 / 실행한 확인 명령과 결과 / 남은 제약")에 다음을 추가 권장:

- **수동 GUI 확인 결과**: 9단계 중 화면에서 직접 본 단계와 미확인 단계.
- **스모크 stdout 발췌**: 사건 1·2·3의 결과 문구와 세대 요약 첫 줄.

### `--quit-after 2` 처리

삭제 또는 교체. `--smoke` 분기에서 `GetTree().Quit()`을 직접 호출하면 별도 종료 옵션이 필요 없다.
