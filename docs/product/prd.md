# LegacyLog PRD

> 최종 수정: 2026-05-18
> 상태: 초안 - MVP 첫 루프, 시작 전제, 사건 결과 피드백, 첫 루프 제품 기준 확정

## 1. 제품 목표

LegacyLog는 가문 연대기 로그라이크 게임이다.

제품 측면의 목표는 플레이어가 한 인물이 아니라 여러 세대에 걸친 가문 기록을 만들고, 선택과 사건의 누적 결과를 연대기로 확인하는 경험을 제공하는 것이다.

프로젝트 운영 측면의 목표는 루트 PRD와 하위 Godot 구현 프로젝트를 분리해, 제품 요구와 구현 설계를 명확히 나누는 것이다.

## 2. 대상 사용자

- 짧은 플레이 세션 안에서 선택의 결과가 누적되는 전략/로그라이크 경험을 원하는 플레이어
- 캐릭터 개인보다 가문, 계보, 연대기, 사건 기록에 관심이 있는 플레이어

## 3. MVP 방향

MVP의 첫 플레이 루프는 **사건 선택 중심**으로 한다. 자세한 결정은 `docs/decisions/003-mvp-first-loop.md`와 `docs/decisions/006-mvp-first-loop-product-criteria.md`를 따른다.

플레이어는 짧은 실행 안에서 다음 흐름을 반복한다.

1. 고정된 첫 가문과 초대 인물로 시작한다.
2. 사건을 읽는다.
3. 선택지 중 하나를 고른다.
4. 선택 결과 문구를 확인한다.
5. 선택 결과가 최소 상태 변화로 반영된다.
6. 결과가 연대기 항목으로 기록된다.
7. 3개의 사건이 끝나면 이번 세대의 요약을 확인한다.

연대기 생성은 별도 장르가 아니라 사건 선택 루프의 핵심 출력으로 다룬다. MVP는 "사건 선택 게임"이지만, 결과 화면은 단순 디버그 로그가 아니라 가문사처럼 읽히는 기록이어야 한다.

## 4. MVP 범위

### 포함

- 새 실행 시작
- 고정된 첫 가문과 초대 인물의 최소 시작 상태
- 3개의 사건 표시
- 사건별 선택지 표시
- 선택 결과 적용
- 현재 상태 표시
- 선택 결과의 연대기 기록
- 선택 결과의 최소 상태 변화 표시
- 3개의 사건 이후 세대 요약 표시

### 제외

- 완성형 가문 생성기
- 복잡한 후계/혼인/혈통 시스템
- 정교한 자원 경제
- 대량 사건 콘텐츠
- 저장/불러오기
- 완성형 UI/아트 스타일 확정
- 밸런스 수치 확정
- 최종 콘텐츠 문안 확정
- CI, 배포, 플랫폼 패키징

## 5. 핵심 경험 요구사항

- 플레이어는 사건마다 의미 있는 선택을 한다.
- 선택은 결과 문구, 현재 상태, 연대기 기록에 영향을 준다.
- 게임은 단일 캐릭터의 생존보다 가문의 지속성과 기록을 중심에 둔다.
- 연대기는 단순 로그가 아니라 플레이어가 만든 가문사의 요약으로 읽혀야 한다.
- 플레이어는 선택 직후 자신의 선택이 어떤 결과를 만들었는지 이해할 수 있어야 한다.
- MVP의 재미 판단은 "다음 사건을 보고 싶다"와 "내 선택이 기록으로 남았다"는 감각을 기준으로 한다.
- MVP에서 의미 있는 선택은 선택 후 결과 문구와 연대기 기록이 구분되고, 플레이어가 볼 수 있는 상태 변화 차이를 만든다는 뜻이다.

## 6. 기능 요구사항

### 시작 흐름

- 플레이어는 새 실행을 시작할 수 있다.
- MVP에서는 복잡한 생성 UI를 요구하지 않는다.
- MVP는 고정된 첫 가문과 초대 인물로 시작한다.
- 플레이어는 복잡한 생성 절차 없이 첫 사건으로 진입한다.
- 선택 결과는 초대 인물의 행적이자 가문의 첫 연대기로 기록된다.
- 시작 상태는 첫 사건 루프를 진행할 수 있을 만큼만 제공한다.

### 사건 흐름

- 게임은 3개의 사건을 순서대로 진행할 수 있어야 한다.
- 게임은 현재 사건의 제목 또는 본문을 보여준다.
- 각 사건은 둘 이상의 선택지를 제공한다.
- 플레이어가 선택지를 고르면 결과 문구, 연대기 항목, 최소 상태 변화를 확인할 수 있다.
- 결과 확인 후 다음 사건으로 진행할 수 있다.
- 각 선택지는 서로 구분되는 결과 문구와 연대기 기록을 가져야 한다.
- 각 선택지는 선택의 차이를 이해할 수 있는 최소 상태 변화 표시를 가져야 한다.

### 상태 모델

- 게임은 현재 실행의 상태를 가진다.
- 상태는 선택 결과로 바뀔 수 있어야 한다.
- 플레이어는 현재 상태 항목을 볼 수 있어야 한다.
- 상태 변화는 플레이어가 선택의 영향을 이해할 수 있을 만큼만 표시한다.
- 선택 결과는 적어도 하나의 상태 항목 변화를 표시해야 한다.
- MVP에서는 정확한 밸런스 수치보다 "선택 결과가 상태에 반영된다"는 제품 요구를 우선한다.
- 구체적인 상태 값 이름, 개수, 데이터 구조, 구현 방식은 Project Codex와 Project Claude의 하위 흐름에서 결정한다.

### 연대기

- 선택 결과는 연대기 항목으로 누적된다.
- 연대기 항목은 사건, 선택, 결과를 플레이어가 이해할 수 있는 문장으로 보여줘야 한다.
- 세대 요약은 지금까지의 연대기를 바탕으로 이번 실행을 짧게 되돌아보게 해야 한다.

### 종료/요약

- MVP는 3개의 사건이 끝나면 세대 요약 화면으로 이동한다.
- 세대 요약은 누적된 연대기 항목, 최종 상태 요약, 이번 세대를 되돌아보는 짧은 마무리 문장을 포함해야 한다.
- 가문 단절, 장기 목표 달성, 다세대 승계는 MVP 이후로 미룬다.

## 7. MVP 완료 기준

MVP는 다음 조건을 만족하면 완료로 본다.

- 새 실행을 시작할 수 있다.
- 플레이어가 사건을 읽고 선택지를 고를 수 있다.
- 선택 후 결과 문구와 최소 상태 변화를 확인할 수 있다.
- 선택 결과가 연대기에 누적된다.
- 3개의 사건 후 세대 요약에서 누적 연대기, 최종 상태 요약, 짧은 마무리 문장을 볼 수 있다.
- Project Claude가 지정된 확인 명령을 실행하고 결과를 보고할 수 있다.

## 8. 구현 연결 방식

루트 폴더는 PRD와 현재 초점을 관리한다. 하위 `legacy-log-project/`는 실제 Godot 구현을 담당한다. 기본 흐름은 `docs/decisions/002-agent-workflow.md`를 따른다.

PRD는 제품 요구 문서다. PRD는 플레이어 경험, MVP 범위, 제외 범위, 제품 완료 기준을 정한다. 구체적인 아키텍처, 구현 설계, 데이터 저장 형식, 코드 구조, 파일 경로, Godot 씬/리소스 구성은 Project Codex와 Project Claude의 하위 흐름에 위임한다.

구현 세션에 작업을 넘길 때는 Root Codex가 `docs/current-focus.md`에 다음을 적는다.

- 참조할 PRD 섹션
- 목적
- 사용자 경험 범위
- 제외 범위
- 사전 가정
- 제품 완료 기준
- 보고 형식

이후 하위 흐름은 다음 순서로 진행한다.

1. Project Codex가 작업 지시를 기준으로 아키텍처 및 구현 계획을 작성한다.
2. Project Claude가 구현 계획을 검토하고 개선점을 제안한다.
3. Project Codex가 개선점을 검토한 뒤 최종 작업 지시서를 작성한다.
4. Project Claude가 최종 작업 지시서에 따라 구현한다.

하위 구현 에이전트는 `docs/current-focus.md`와 최종 작업 지시서에서 지정한 범위만 구현 대상으로 삼는다.

## 9. 미결정 사항

- 사건 데이터의 구체적 저장 형식
- 상태 값의 이름과 개수
- 첫 가문과 초대 인물의 정확한 이름 및 도입 문장
- 첫 MVP 사건들의 실제 문안
- 세대 요약 화면의 최종 UI 형식
