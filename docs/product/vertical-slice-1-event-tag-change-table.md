# Vertical Slice 1 사건별 유산 태그 변화 표

> 최종 수정: 2026-05-19

## 목적

이 문서는 Vertical Slice 1 사건 46개의 선택 결과가 어떤 유산 태그를 추가, 해소, 교체할 수 있는지 정리한다.

목표는 구현 데이터 구조를 정하는 것이 아니라, 플레이어 선택이 다음 세대와 이후 사건 풀에 남는 방식을 Root 기획 기준으로 고정하는 것이다.

## 적용 범위

- `docs/product/vertical-slice-1-event-list.md`의 46개 사건
- 선택지별 유산 태그 추가 후보
- 선택지별 유산 태그 해소 또는 교체 후보
- 태그 변화가 UI에 노출되어야 하는 경우

## 제외 범위

- 태그 부여 확률
- 태그 지속 턴 수의 구현 방식
- 태그 저장 데이터 구조
- 상태 수치 변화량
- 선택지 성공/실패 판정 공식

## 작성 원칙

- 큰 사건은 0~2개의 유산 태그를 남길 수 있다.
- 보조 사건은 0~1개의 유산 태그를 남길 수 있다.
- 상반 태그는 동시에 유지하지 않고 교체를 우선한다.
- 태그 변화가 다음 세대 사건 조건에 영향을 주면 결과 화면에서 반드시 보여준다.
- 태그가 붙지 않는 선택지도 허용한다. 모든 선택이 유산 태그를 남기면 피로도가 커진다.

## UI/UX 노출 기준

태그 변화는 결과 화면에서 다음 우선순위로 노출한다.

1. 새로 추가된 가문사 태그
2. 새로 추가된 세대 태그
3. 상반 태그 교체
4. 해소된 부정 태그
5. 단기 태그 추가

사건 결과 화면에서는 모든 태그를 한꺼번에 나열하지 않는다. 해당 선택으로 변한 태그만 보여준다.

세대 시작 화면에서는 이번 세대 사건 풀에 영향을 주는 태그만 우선 노출한다.

## 수동 검증 질문

- 선택 결과에 붙은 태그가 납득되는가?
- 태그 변화가 너무 자주 발생해 피로하지 않은가?
- 부정 태그 해소가 명확한 보상처럼 느껴지는가?
- 상반 태그가 동시에 유지되지 않고 교체되는가?
- 태그가 다음 세대 사건 등장 이유로 보이는가?
- 결과 화면에서 태그 변화가 필요한 만큼만 보이는가?
