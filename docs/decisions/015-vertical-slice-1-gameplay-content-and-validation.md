# 015. Vertical Slice 1 콘텐츠·작위·UI 검증 기준

## 상태
확정 (2026-05-19)

## 맥락

`docs/decisions/014-vertical-slice-1-playtest-scope.md`에서 Vertical Slice 1은 중세 판타지 귀족가문 반복 플레이 재미 검증 빌드로 정의되었다.

이번 결정은 그 검증 빌드가 실제로 다양한 플레이 양상을 만들 수 있도록 이벤트 분량, 작위 상승/상실, 유산 태그, 후계자 차이, 세대 시작 정보 구조, UI 우선순위, 반복 플레이 피드백 형식을 정한다.

구체적인 사건 본문, 수치, 데이터 파일 형식, 구현 구조, 밸런스 값은 이 문서에서 확정하지 않는다.

## 결정

### 이벤트 분류별 최소 개수

Vertical Slice 1의 이벤트는 **최소 46개**를 확정 기준으로 두고, 가능하면 **60개 내외**까지 확장한다.

최소 구성은 다음과 같다.

| 분류 | 최소 개수 | 역할 |
|---|---:|---|
| 영지/재산 사건 | 8 | 재산, 유지비, 영지 경영, 세금, 흉작/풍작 |
| 명예/전쟁/기사도 사건 | 8 | 전공, 기사도, 결투, 전장 명성 |
| 궁정/정치 사건 | 8 | 왕의 총애, 파벌, 궁정 음모, 청원 |
| 가문 내부/후계 사건 | 8 | 후계 안정, 친족 갈등, 교육, 분가 |
| 작위 상승/청원/포상 사건 | 5 | 승격 기회, 왕명, 공훈 보상 |
| 작위 상실/위기/반역 의심 사건 | 5 | 박탈, 반역 의심, 정적 공격, 몰락 압박 |
| 세대 종결 변형 사건 | 4 | 자연사, 전사, 병사, 폐위/추방 |

각 이벤트는 가능하면 다음 조건 중 하나 이상을 가진다.

- 작위 조건
- 상태 조건
- 유산 태그 조건
- 세대 또는 장기 진행 조건

각 이벤트 결과는 다음 중 하나 이상을 남겨야 한다.

- 상태 변화
- 연대기 항목
- 유산 태그
- 다음 세대 시작 조건 변화
- 작위 상승/상실 가능성 변화

### 작위 상승/상실 조건과 비용

작위 상승은 자동 승급이 아니라 **조건 충족 후 등장하는 승격 기회 이벤트**로 처리한다.

승격 기회는 현재 작위, 상태, 유산 태그, 궁정 영향력, 명예, 장기 진행 조건에 따라 등장할 수 있다.

기본 방향은 다음과 같다.

- 남작 -> 자작: 재산 또는 명예가 충분하고 영지 기반이 안정적일 때
- 자작 -> 백작: 궁정 영향력 또는 전쟁 공훈이 높을 때
- 백작 -> 후작: 변경 방위, 군사적 공헌, 왕실 신뢰가 있을 때
- 후작 -> 공작: 장기적 권력 기반과 왕권 관계가 충분할 때

승격은 자동 보상이 아니라 선택이다. 승격에는 비용 또는 위험이 따른다.

가능한 비용과 위험은 다음과 같다.

- 재산 소모
- 가문 결속 하락
- 궁정 정적 증가
- 명예 손상 가능성
- 다음 세대 위험 태그 부여

작위 상실은 즉시 패배가 아니라 **멸문 전 쇠락 단계**로 처리한다.

작위 상실은 다음 방식으로 표현할 수 있다.

- 대표작위 상실
- 하위 작위만 유지
- 특정 영지 상실
- 작위 유지 비용 증가
- 후계 권리 약화
- 멸문 위험 증가

작위는 높을수록 더 큰 기회와 더 큰 위험을 동시에 열어야 한다.

### 유산 태그

Vertical Slice 1은 유산 태그를 세대 간 이어짐의 핵심 매개로 사용한다.

유산 태그는 다음 세 가지 역할을 한다.

- 다음 세대 사건 조건을 연다.
- 세대 시작 상황을 바꾼다.
- 연대기에서 가문의 정체성을 만든다.

초기 태그 계열과 후보는 다음과 같다.

| 계열 | 후보 |
|---|---|
| 왕권/궁정 | 왕의 총애, 궁정의 의심, 정적의 표적, 왕실의 채무 |
| 명예/불명예 | 전쟁 공훈, 기사의 명예, 불명예, 겁쟁이의 소문 |
| 영지/재산 | 풍요로운 영지, 피폐한 영지, 상인 후원, 가문의 부채 |
| 가문 내부/후계 | 안정된 후계, 불안한 후계, 내부 균열, 충성스러운 친족 |
| 작위/정치적 입지 | 승격 명분, 불안한 승격, 박탈 위기, 변경의 책무 |
| 비극/몰락 | 피의 원한, 잊히지 않는 패배, 가문의 맹세, 몰락의 징조 |

큰 사건은 0~2개, 작은 사건은 0~1개의 유산 태그를 남길 수 있다.

상반되는 태그는 동시에 오래 유지하지 않는 방향을 권장한다. 태그의 지속 기간, 제거 규칙, 충돌 처리 방식은 구현과 밸런스 단계에서 조정한다.

태그는 UI에서 조건부 사건과 세대 변화의 이유로 노출되어야 한다.

### 후계자 성향·강점·약점

Vertical Slice 1은 후계자에게 **성향, 강점, 약점**을 부여한다.

목적은 다음 세대가 이전 세대와 다르게 느껴지게 하고, 후계 선택이 단순 진행 버튼이 아니라 다음 세대 방향을 가늠하는 판단이 되게 하는 것이다.

성향 후보는 다음 6개 내외로 시작한다.

- 신중함
- 야심
- 명예 중시
- 실리적
- 가문 중심
- 궁정 지향

강점 후보는 다음 풀에서 시작한다.

- 영지 경영
- 전장 지휘
- 궁정 처세
- 가문 중재
- 재정 감각
- 기사도 평판
- 청원 능력
- 위기 수습
- 후계 교육
- 파벌 교섭

약점 후보는 다음 풀에서 시작한다.

- 낭비벽
- 성급함
- 고립
- 허약함
- 의심받는 혈통
- 오만함
- 친족 불화
- 전쟁 기피
- 궁정 미숙
- 빚에 취약

성향, 강점, 약점은 사건 등장 조건, 선택 결과 보정, 위험 사건 조건에 약하게 영향을 준다.

후계자 특성은 절대 성공/실패를 단독 결정하지 않는다. 선택과 가문 상태가 더 중요해야 한다.

후계 후보는 기본 1명으로 두되, 조건에 따라 2명 이상 또는 후계 없음이 가능하다.

후계 화면은 후보별로 다음을 간결하게 보여준다.

- 이름
- 성향
- 강점
- 약점
- 승계 사유 한 줄

### 세대 시작 화면 정보 구조

세대 시작 화면은 이전 세대 결과와 이번 세대 조건을 연결하는 화면이다.

목표는 **다음 세대가 왜 이 상태에서 시작하는지 10초 안에 이해시키는 것**이다.

기본 구성은 다음 다섯 블록으로 한다.

1. 가문 헤더
2. 이전 세대의 유산
3. 이번 세대의 출발 상태
4. 후계자 소개
5. 이번 세대의 압력 또는 예고

표시량 기준은 다음과 같다.

- 이전 세대 핵심 요약: 1~2문장
- 유산 태그: 2~4개
- 상태 라벨: 핵심 상태 4개
- 후계 정보: 이름, 성향, 강점, 약점, 승계 사유
- 이번 세대 압력: 1~2문장

세대 요약은 "지난 세대가 무엇으로 기억되는가"를 다루고, 세대 시작 화면은 "그 결과 이번 세대는 어떤 조건에서 시작하는가"를 다룬다.

세부 로그는 가문사 또는 연대기 탭에서 확인하게 한다.

### 작위 위험과 상태 변화 UI 우선순위

UI의 핵심 원칙은 **플레이어가 선택 직전과 선택 직후에 알아야 할 정보만 전면에 두는 것**이다.

항상 보이는 정보는 다음으로 제한한다.

- 가문명
- 대표작위
- 세대
- 현재 인물
- 현재 단계

가장 중요한 위험 경고 1개는 선택적으로 헤더 또는 상단에 표시할 수 있다.

사건 화면은 다음을 보여줘야 한다.

- 사건 분류
- 관련 상태
- 관련 작위 위험
- 관련 유산 태그

선택지는 **짧은 행동명 + 영향 요약**으로 구성한다. 긴 설명이나 정확한 숫자는 상세 영역으로 보낸다.

결과 화면은 다음 순서를 따른다.

1. 결과 문장
2. 상태 변화 요약
3. 태그 변화
4. 작위 변화 또는 작위 위험 변화
5. 연대기 기록
6. 다음 행동

정확한 숫자, 전체 태그 목록, 작위 조건 상세는 상태 탭, 가문사 탭, 상세 토글 등 보조 영역에 둔다.

작위 위험은 수치보다 단계 라벨로 표시한다.

### 반복 플레이 피드백 기록 형식

Vertical Slice 1은 최소 3회, 가능하면 5회 반복 플레이로 검증한다.

회차별 기록은 다음 항목을 포함한다.

- 실행 결과: 도달 세대, 최종 대표작위, 종료 유형, 주요 가문 궤적
- 기억에 남은 선택: 선택, 고민한 이유, 결과 납득 여부, 다시 선택할지 여부
- 작위 체감: 상승 기대, 높은 작위의 부담, 작위 상실/위험 이해 여부
- 세대 간 이어짐: 이전 세대 흔적, 유산 태그 체감, 후계자 차이
- UI/UX: 현재 상황 이해, 선택 영향 확인, 정보 탐색 난이도, 작은 창/세로형 불편, 가장 헷갈린 화면
- 종합: 재미 점수 1~5, 재플레이 의향, 가장 큰 개선 필요점

3~5회 플레이 후에는 전체 비교 기록을 남긴다.

- 다양성: 회차마다 다른 전개가 나왔는지, 같은 선택 반복 여부, 이벤트 중복 체감
- 전략 차이: 상승 전략, 안정 전략, 위험 감수 전략의 차이
- 가문사 감각: 쌓이고 무너지는 느낌, 납득 가능한 멸문/쇠락, 기억에 남는 이야기
- UI/UX 종합: 자주 확인한 정보, 찾기 어려운 정보, 과하게 많이 보인 정보, 반드시 개선할 UI
- 다음 판단: 재미 검증 통과/보류/실패, 이유, 다음 구현 우선순위

재미 점수는 다음 기준을 따른다.

- 1점: 다시 하고 싶지 않음
- 2점: 구조는 보이나 지루함
- 3점: 가능성은 있음, 개선 필요
- 4점: 다시 해보고 싶음
- 5점: 다른 전략으로 여러 번 해보고 싶음

피드백은 기능 문제, UI 문제, 기획 문제, 콘텐츠 문제를 구분해 기록한다.

## 근거

Vertical Slice 1은 기능 존재 여부보다 반복 플레이에서 다른 가문사가 나오는지를 검증해야 한다. 그러려면 충분한 이벤트 분량, 조건부 등장, 유산 태그, 후계자 차이, 작위 위험이 함께 필요하다.

작위 상승이 자동 보상이 되면 게임이 단순 레벨업이 된다. 승격 기회와 비용, 작위 상실 위험을 함께 두면 플레이어는 "올라갈 수 있는가"뿐 아니라 "지금 올라가도 버틸 수 있는가"를 판단하게 된다.

UI/UX는 이 빌드의 재미 검증 조건이다. 정보가 많아질수록 플레이어가 선택의 원인과 결과를 이해하지 못하면 게임성이 검증되지 않는다.

피드백 기록 형식을 미리 정하면 감상문이 아니라 비교 가능한 검증 자료를 얻을 수 있다.

## 영향

`docs/product/prd.md`의 Vertical Slice 1 절은 이 문서의 콘텐츠·작위·UI 검증 기준을 참조해야 한다.

`docs/current-focus.md`의 미정 질문은 이 문서로 대부분 해소된다. 다음 초점은 Root Codex의 세부 콘텐츠 기획 또는 Project Codex 구현 위임 준비로 옮길 수 있다.

하위 Project Codex로 구현을 넘길 때는 이 문서의 이벤트 분류, 작위 구조, 유산 태그, 후계자 정보, 세대 시작 화면, UI 우선순위, 피드백 양식을 함께 포함해야 한다.

## 제외 범위

- 데이터 파일 형식과 로딩 구조 결정
- 이름 데이터 실제 작성
- Godot 씬, C# 클래스, 리소스 구조 결정
- 최종 UI 레이아웃과 아트 스타일 확정

## 결정 보류

결정 보류 — 콘텐츠 상세 — Root 후속 기획 라운드 위임: 사건별 ID, 본문, 선택지, 결과 문장, 연대기 문안은 이 문서에서 확정하지 않는다.

결정 보류 — 밸런스 — Root 후속 기획 라운드 위임: 상태 변화량, 작위 상승/상실 조건의 정확한 수치, 태그 지속 기간, 이벤트 등장 가중치는 이 문서에서 확정하지 않는다.

결정 보류 — 기술 구조 — 하위 구현 흐름 위임: 이벤트 데이터 파일 형식, 태그 모델, 후계자 특성 모델, UI 레이아웃 구현 방식은 Project Codex와 Project Claude 흐름에서 결정한다.
