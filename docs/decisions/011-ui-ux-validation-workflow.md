# 011. 기능 검증과 UI/UX 수동 검증 병행

## 상태
확정 (2026-05-19)

## 맥락

다세대 MVP 구현은 자동 스모크와 빌드 기준으로 동작을 확인할 수 있다. 그러나 LegacyLog는 연대기, 세대 요약, 후계, 멸문, 상태 표시를 플레이어가 읽고 조작하며 이해해야 하는 게임이다.

따라서 기능이 동작한다는 사실만으로는 제품 완료를 판단하기 어렵다. 특히 텍스트 밀도, 화면 전환, 버튼 흐름, 상태 정보의 이해 가능성, 세대 간 이어짐의 체감은 사용자가 직접 플레이하면서 확인해야 한다.

## 결정

모든 구현 작업은 기능 검증과 UI/UX 검증을 함께 고려한다.

Project Codex가 구현 계획 또는 최종 작업 지시서를 작성할 때는 각 작업에 대해 다음을 명시한다.

- 작업 의도: 이 변경이 플레이어 경험에서 무엇을 개선하거나 확인하려는 것인지
- 사용자 조작 흐름: 사용자가 실제로 어떤 순서로 보고 누르고 판단하는지
- 기능 완료 기준: 자동 검증, 빌드, 스모크 등으로 확인할 수 있는 조건
- UI/UX 수동 검증 기준: 사용자가 직접 플레이하며 확인해야 하는 조건
- 완료 판단 분리: 기능 동작 완료와 UI/UX 확인 완료를 구분해 보고하는 방식

Project Claude가 구현 결과를 보고할 때는 다음을 구분한다.

- 자동 검증 결과
- 수동 GUI 또는 플레이 확인 결과
- 수동 확인을 하지 못한 경우 그 이유
- UI/UX상 관찰된 문제 또는 확인 보류 항목

Root Codex는 진행상황을 판단할 때 기능 구현 완료와 제품 경험 완료를 분리한다. 기능이 동작해도 수동 UI/UX 검증이 부족하면 "기능 구현 완료, UI/UX 검증 필요" 상태로 둔다.

사용자에게 수동 검증을 요청할 때는 다음을 함께 제공한다.

- 현재 구현된 것
- 사용자가 직접 테스트해야 할 흐름
- 화면별 확인 항목
- 피드백 기록 형식
- 개선 작업으로 되돌릴 기준

## UI/UX 수동 검증 기본 항목

각 구현 작업의 수동 검증 항목은 작업 성격에 맞게 조정하되, 기본적으로 다음을 확인한다.

- 사용자가 현재 화면의 목적을 즉시 이해할 수 있는가
- 다음에 눌러야 할 주요 행동이 명확한가
- 상태, 결과, 연대기, 세대 요약이 서로 헷갈리지 않는가
- 텍스트가 과밀하거나 잘리지 않는가
- 화면 전환 뒤 이전 선택의 결과가 이어졌다는 점이 보이는가
- 실패, 종료, 멸문 같은 부정적 결과도 명확하고 납득 가능하게 전달되는가
- 상세 정보는 필요한 사용자가 찾을 수 있고, 기본 흐름을 방해하지 않는가
- 수동 플레이 후 사용자가 불편을 느낀 지점이 보고 형식에 남는가

## 근거

자동 검증은 흐름이 끊기지 않는지 확인하는 데 강하다. 그러나 화면을 읽고 이해하는 부담, 버튼 흐름의 자연스러움, 텍스트 밀도, 플레이어가 느끼는 의미감은 자동 검증만으로 판단하기 어렵다.

LegacyLog의 핵심 원칙은 랜덤성과 부분적 유저 주도성 위에서 선택이 사회적·서사적으로 의미 있었다고 느끼게 하는 것이다. 이 감각은 기능 동작만으로 보장되지 않으며, 실제 플레이 조작과 UI/UX 검증이 함께 필요하다.

## 영향

`docs/decisions/002-agent-workflow.md`의 멀티 에이전트 흐름을 유지하되, Project Codex의 작업 지시서와 Project Claude의 구현 보고에는 UI/UX 검증 기준이 포함되어야 한다.

`docs/current-focus.md`의 진행상황 판단은 기능 구현 완료와 UI/UX 수동 검증 완료를 구분해야 한다.

향후 구현 위임 프롬프트는 기능 검증 명령만 적지 않고, 사용자가 직접 플레이하며 확인할 UI/UX 체크리스트도 포함해야 한다.

사용자 수동 검증 안내는 `docs/manual-ui-ux-validation-guide.md`처럼 별도 문서로 제공할 수 있다.
