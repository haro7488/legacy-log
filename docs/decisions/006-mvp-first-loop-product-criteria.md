# 006. MVP 첫 루프 제품 기준

## 상태
확정 (2026-05-18)

## 맥락

MVP 첫 플레이 루프, 시작 전제, 사건 결과 피드백은 이미 확정되었다. 하지만 실제 첫 루프를 구현 흐름으로 넘기려면 사건 수, 사건 구성, 의미 있는 선택, 상태 피드백, 세대 요약의 최소 제품 기준이 더 필요하다.

이번 결정은 구현 세부 설계가 아니라 플레이어가 관찰할 수 있는 첫 루프의 제품 완료 기준을 좁히기 위한 것이다.

## 결정

MVP 첫 루프는 **3개의 사건**으로 구성한다.

각 사건은 다음을 포함해야 한다.

- 사건 제목 또는 짧은 본문
- 선택지 2개 이상
- 선택지별 결과 문구
- 선택지별 연대기 기록 문장
- 선택지별 최소 상태 변화 표시

MVP에서 의미 있는 선택은 다음 조건을 만족해야 한다.

- 선택 후 결과 문구가 구분된다.
- 연대기에 남는 기록 문장이 구분된다.
- 플레이어가 볼 수 있는 상태 변화 차이를 만든다.
- 플레이어가 선택 직후 차이를 이해할 수 있다.

상태 모델은 다음 제품 기준만 요구한다.

- 플레이어가 현재 상태 항목을 볼 수 있어야 한다.
- 선택 결과는 적어도 하나의 상태 항목 변화로 표시되어야 한다.
- 상태 값 이름, 개수, 수치 범위, 데이터 구조, 구현 방식은 하위 구현 흐름에서 정한다.

세대 요약은 다음을 포함해야 한다.

- 이번 루프에서 누적된 연대기 항목
- 최종 상태 요약
- 이번 세대를 되돌아보는 짧은 마무리 문장

## 의도적으로 확정하지 않는 것

- 상태 값의 실제 이름과 개수
- 상태 수치와 밸런스
- 첫 가문과 초대 인물의 실제 이름
- 첫 MVP 사건들의 최종 문안
- 세대 요약 화면의 최종 UI 형식
- 사건 데이터 저장 형식과 구현 구조

## 근거

3개의 사건은 선택 결과가 누적되고 세대 요약이 의미를 가지는 최소량이다. 1개나 2개 사건은 연대기 누적 감각이 약하고, 5개 이상은 첫 구현 전에 콘텐츠 부담이 커진다.

의미 있는 선택 기준은 단순히 다른 버튼을 누르는 것이 아니라, 결과 문구와 연대기, 상태 피드백이 함께 달라지는 경험을 보장하기 위한 것이다.

상태 모델은 첫 루프 검증에 필요한 관찰 가능한 피드백만 요구한다. 구체적인 이름, 수치, 데이터 구조는 하위 구현 흐름에서 결정하도록 남겨 구현 설계 영역을 침범하지 않는다.
