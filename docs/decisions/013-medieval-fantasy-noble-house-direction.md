# 013. 중세 판타지 귀족가문 방향

## 상태
확정 (2026-05-19)

## 맥락

LegacyLog는 다세대 가문 연대기 게임으로 정체가 확정되었다. 이후 콘텐츠와 상태 라벨, 이름 데이터, 작위 표현을 정리하려면 가문이 어떤 사회적 위치에 있는지 먼저 정해야 한다.

이번 결정은 세계관과 작위, 이름 기준을 제품 방향 수준에서 정리한다. 구체적인 데이터 파일 형식, 생성 알고리즘, Godot 리소스 구조, C# 타입 설계는 하위 Project Codex 흐름에 위임한다.

## 결정

### 세계관 방향

LegacyLog의 기본 세계관 방향은 **중세 판타지**로 한다.

플레이어가 시뮬레이션하는 가문은 일반 가문이 아니라 **귀족가문**이다. 게임의 핵심 흐름은 귀족가문이 작위, 영지, 평판, 후계, 사건의 누적 결과를 통해 이어지거나 몰락하는 과정이다.

### 작위 체계

기본 작위 체계는 유럽 중세 귀족 계급도를 게임용으로 단순화해 사용한다.

기본 단계는 다음 순서를 따른다.

1. 남작
2. 자작
3. 백작
4. 후작
5. 공작
6. 왕
7. 황제

MVP 이후 확장에서는 한 가문이 여러 작위를 동시에 가질 수 있다. 작위는 단일 등급 값이 아니라, 가문이 보유한 사회적 권리와 위상의 묶음으로 취급할 수 있다.

가문이 여러 작위를 보유한 경우, 플레이어 또는 시스템은 보유 작위 중 가장 상위 작위군에서 **대표작위**를 정할 수 있어야 한다. 대표작위는 UI, 연대기, 세대 요약에서 가문을 부르는 기본 호칭으로 쓰일 수 있다.

### 시작 작위

기본 시작점은 **최소 남작위**로 한다.

MVP 또는 초기 구현에서 초대 가문은 남작급 귀족가문으로 시작한다. 더 낮은 평민, 기사, 향사 단계에서 귀족으로 상승하는 플레이는 현재 핵심 방향이 아니다.

### 이름 기준

가문명, 인물명, 영지명 등 이름은 **중세 판타지풍 이름**을 차용한다.

이름은 실제 역사 인물이나 특정 실존 가문을 직접 재현하기보다, 중세 판타지 귀족가문으로 읽히는 조합 규칙과 이름 풀을 사용한다.

### 데이터 관리 책임

PRD는 필요한 이름·작위·세계관 데이터의 **목록과 제품 요구**를 관리한다.

이름 데이터의 기획 기준과 필요한 목록은 Root Codex가 정한다. 실제 파일 형식, 생성 규칙 구현, 중복 방지 방식, 로컬라이징 구조, Godot 리소스화 여부는 Project Codex와 Project Claude의 하위 구현 흐름에서 결정한다.

## 근거

귀족가문 시뮬레이션으로 방향을 좁히면 다세대 진행, 후계, 멸문, 작위 상승과 몰락, 연대기 기록이 하나의 제품 정체로 묶인다.

남작부터 황제까지의 단순화된 계급도는 플레이어가 사회적 상승과 몰락을 직관적으로 이해하게 한다. 실제 중세의 복잡한 봉건 구조를 그대로 재현하면 MVP 범위와 UI 설명 부담이 커진다.

여러 작위와 대표작위 개념은 귀족가문의 장기 진행을 지원한다. 한 가문이 세대마다 다른 영지와 권리를 얻거나 잃더라도, 플레이어는 대표작위를 통해 현재 가문의 사회적 얼굴을 이해할 수 있다.

이름 데이터의 목록은 PRD에서 관리하되, 실제 데이터 구조를 루트에서 확정하지 않는 것은 `docs/decisions/002-agent-workflow.md`의 역할 분담과 맞다.

## 영향

`docs/product/prd.md`는 LegacyLog를 중세 판타지 귀족가문 시뮬레이션으로 설명해야 한다.

상태 라벨, 사건 문안, 연대기 문장, 가문/인물 이름 후보는 이 결정과 충돌하지 않아야 한다.

`docs/decisions/009-chronicle-style.md`에서 보류한 시대/세계관 톤은 이 문서로 일부 해소된다. 다만 진행 중 1인칭, 후대 기록자 시점, 사관식 서정·중립 톤 원칙은 그대로 유지된다.

`docs/decisions/010-state-display-layering.md`에서 보류한 상태 구간 라벨 문구는 중세 판타지 귀족가문 톤에 맞춰 후속 라운드에서 정한다.

하위 구현 흐름은 Root가 정한 이름과 작위 기획 기준을 구현 가능한 데이터 구조와 알고리즘으로 옮긴다. 구체적인 데이터 구조와 알고리즘은 별도 Project Codex 계획을 거친 뒤 결정한다.

## 결정 보류

결정 보류 — 문체 세부 — 별도 기획 라운드 위임: 중세 판타지 방향은 확정했지만, 진행 중 1인칭 문장의 어휘 밀도, 후대 기록자의 최종 문체, 사건별 문장 톤은 아직 확정하지 않는다.

결정 보류 — 이름 데이터 기획 — Root 후속 기획 라운드 위임: 이름 풀의 문화권 세분화, 이름 후보 목록, 가문명/인물명/영지명 작성 기준은 이 문서에서 정하지 않는다.

결정 보류 — 이름 데이터 구현 — 하위 구현 흐름 위임: 이름 풀의 파일 형식, 생성 알고리즘, 중복 방지, 데이터 검증 방식은 이 문서에서 정하지 않는다.

결정 보류 — 게임 요소 구체화 — 별도 기획 라운드 위임: 작위가 어떤 상태, 보상, 위험, 승리/패배 조건과 연결되는지는 후속 게임 요소 정리에서 다룬다.
