# 012. 반응형 탭 기반 UI/UX 개선 방향

## 상태
확정 (2026-05-19)

## 맥락

다세대 MVP 기능은 구현 보고 기준으로 동작한다. 그러나 현재 UI는 기능 경로를 확인하기 위한 성격이 강하며, 수동 검증 전에 플레이어가 실제로 불편 없이 조작할 수 있는 UI 구조로 정리할 필요가 있다.

LegacyLog는 텍스트, 상태, 연대기, 세대 요약, 후계, 멸문 정보를 함께 다루므로 화면 정보가 쉽게 과밀해질 수 있다. 특히 작은 창과 세로형 모바일 기기에서도 사용할 수 있는 범용 구조가 필요하다.

## 결정

### 대상 화면

UI/UX 개선은 **작은 창과 세로형 모바일 기기 모두에서 사용 가능한 범용 디자인**을 기준으로 한다.

데스크톱 대형 화면만을 전제로 넓은 가로 레이아웃을 만들지 않는다. 우선순위는 좁은 폭에서도 읽고 조작할 수 있는 세로 흐름이다.

### 레이아웃 구조

UI는 이후 변경, 추가, 삭제가 쉽도록 명확한 레이아웃 구조를 가진다.

화면은 최소한 다음 역할로 나뉘어야 한다.

- 현재 맥락 영역: 가문, 세대, 인물, 현재 단계
- 주요 내용 영역: 사건, 결과, 세대 요약, 후계, 멸문 등 현재 화면의 핵심 본문
- 주요 행동 영역: 다음으로 진행하거나 선택하는 버튼
- 보조 정보 영역: 상태, 연대기, 이전 세대 기록, 상세 정보

이 구조는 구현 방식의 세부 아키텍처를 확정하지 않는다. Project Codex와 Project Claude가 Godot UI 구조에 맞게 구체화한다.

### 정보 구조

항상 보여야 하는 정보와 탭에서 확인할 정보를 구분한다.

항상 보여야 하는 정보는 다음 기준으로 정한다.

- 현재 가문
- 현재 세대
- 현재 인물
- 현재 화면의 목적
- 다음 주요 행동

탭 또는 보조 영역에서 확인할 정보는 다음 기준으로 정한다.

- 상태 상세
- 연대기
- 이전 세대 기록
- 이어진 큰 사건 결과 또는 유산
- 특성 스텁

### 탭 방식

기본 UI는 **모바일의 탭 형식으로 오가며 확인할 수 있는 방식**을 기준으로 확장한다.

탭은 화면 정보를 나누기 위한 기본 방식이며, 한 화면에 모든 정보를 동시에 밀어 넣지 않는다. 사용자는 현재 사건/결과 같은 핵심 흐름을 보면서 필요할 때 상태, 연대기, 이력 같은 보조 정보를 탭으로 확인할 수 있어야 한다.

탭 이름과 개수의 제품 기준은 Root Codex가 정한다. Project Codex는 현재 구현과 검증 흐름을 바탕으로 기술적 구현 방식을 제안할 수 있다. 단, 탭이 너무 많아 주요 흐름을 방해하면 안 된다.

### 직접 선택 원칙

구체 UI 구현 방식은 Project Codex와 Project Claude가 플레이 편의성을 기준으로 선택한다.

Root Codex는 정보 우선순위, 화면별 역할, 수동 검증 기준을 정하고, 구체 배치, 컨테이너, Godot Control 구성, 버튼 위치, 토글 구현 방식은 하위 구현 흐름에 위임한다.

### 완료 기준

목표는 사용자가 시작부터 멸문까지 **불편 없이 진행할 수 있는 것**이다.

동시에 체크리스트 기반으로 화면별 문제를 기록할 수 있어야 한다. 따라서 완료 판단은 다음 두 축으로 한다.

- 플레이 흐름 완료: 사용자가 시작부터 멸문까지 막힘 없이 진행할 수 있다.
- 체크리스트 검증: 시작, 사건, 결과, 세대 요약, 후계, 2세대 진입, 멸문, 상태/연대기 탭을 각각 확인했다.

## 근거

작은 창과 세로형 모바일 기기를 기준으로 잡으면 정보 밀도와 버튼 흐름을 더 엄격하게 관리할 수 있다. 넓은 화면에서만 성립하는 UI는 이후 모바일 대응이나 작은 창 검증에서 다시 설계를 뒤집게 만든다.

탭 기반 구조는 다세대 MVP의 정보량을 한 화면에 모두 노출하지 않으면서도, 플레이어가 필요한 정보를 찾아볼 수 있게 한다. 이는 `010-state-display-layering`의 정보 계층화 원칙과도 맞다.

항상 보이는 정보와 탭으로 숨길 정보를 분리하면, 현재 선택과 다음 행동을 방해하지 않으면서 상태와 연대기 같은 보조 정보를 제공할 수 있다.

## 영향

다음 Project Codex 작업은 기능 추가보다 UI/UX 편의성 개선 계획을 우선한다.

Project Codex가 작성할 계획과 Project Claude에게 넘길 지시서에는 다음이 포함되어야 한다.

- 작은 창과 세로형 모바일 대응 기준
- 화면별 주요 목적과 주요 행동
- 항상 보이는 정보와 탭 정보의 구분
- 탭 구조 구현 제안
- 사용자가 직접 플레이하며 확인할 체크리스트
- 기능 검증과 UI/UX 수동 검증의 분리 보고 방식

## 결정 보류

결정 보류 — 구체 UI 구현 방식 — 하위 구현 흐름 위임: Godot Control 노드 구조, 컨테이너 구성, 폰트 크기, 간격, 버튼 위치, 탭 구현 방식은 이 문서에서 정하지 않는다.

결정 보류 — 최종 시각 스타일 — 별도 UI/아트 라운드 위임: 색상, 폰트, 장식, 완성형 아트 스타일은 이 문서에서 정하지 않는다.

결정 보류 — 탭 명칭과 개수 — Root 후속 기획 라운드 위임: 정확한 탭 이름과 탭 수의 제품 기준은 이 문서에서 정하지 않는다.
