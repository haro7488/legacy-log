## 주요 관찰

### 관찰 1: Claude Code의 "판단 유보" 능력
Git 초기화 중간 검증에서 `.claude/settings.local.json`이 예상치 못하게 스테이징 목록에 포함되어 있었다. Claude Code는 이를 자기 판단으로 처리하지 않고 "판단이 필요합니다"라고 멈춰 사용자에게 결정을 넘겼다. 잘 설계된 에이전트는 경계에서 멈추고 물어본다. 이번엔 좋은 쪽이었다.

### 관찰 2: 중간 멈춤 지시의 정확한 준수
`git status` 검증까지만 하고 `git commit` 전에 멈추라는 지시를 정확히 따랐다. 자동으로 다음 단계로 넘어가려는 시도 없음. "되돌리기 비용이 꺾이는 지점 직전에 멈춘다"는 원칙을 지시문으로 명시했을 때 그대로 작동함을 확인.

### 관찰 3: 실패 복구 상황의 제약 준수
원격 `push`가 거부되었을 때(원격에 선행 커밋 존재), 지시문에 명시한 "force push 절대 금지" 제약을 지켰다. 세 가지 복구 선택지(rebase / allow-unrelated-histories / 원격 재생성)를 스스로 제시하고 사용자 결정을 기다림. 실패 시나리오에서도 제약 준수가 유지됨.

### 관찰 4: 기계적 작업에서의 정확성
파일 삭제, `.gitignore` 보강(기존 내용 유지하며 append), 폴더 생성 등 명확한 경계가 있는 작업은 100% 정확하게 수행. "명확한 지시 → 정확한 실행" 가설이 데이터로 확인됨.

### 관찰 5: "특이사항 없음" 보고의 정직성
덩어리 1 보고에서 "특이사항 없음"이라고 정직하게 답함. 억지로 뭔가 만들어내지 않음. 덜 성숙한 에이전트는 "보고할 게 없으면 불안해서" 없는 문제를 만들어내기도 하는데, 그런 경향이 보이지 않음.

## 배운 원칙

### 원칙 1: 되돌리기 비용이 꺾이는 지점 전에 검증한다
**사례:** `git commit` 전에 `git status` 확인. `git push` 전에 `git fetch` 확인.

되돌리기 비용은 작업 종류마다 불연속적으로 커진다. `git init`은 되돌리기 쉽고, `git add`도 쉽지만, `git commit`부터는 번거로워지고, `git push` 후에는 훨씬 더 번거롭다. 이 "꺾이는 지점" 직전에 한 번 멈추는 비용(사람이 30초 출력 읽기)은 잘못된 상태를 되돌리는 비용에 비해 압도적으로 싸다.

이 원칙은 Git뿐만 아니라 **대량 파일 삭제, 이름 일괄 변경, 폴더 구조 대이동, 데이터베이스 변경, 배포** 같은 모든 "비용 곡선이 꺾이는" 작업에 적용할 수 있다.

### 원칙 2: 가정을 명시하고 가정 위반 시 멈춘다
**사례:** 덩어리 3 지시문에 "원격이 비어있다고 가정"을 명시. 실제로 비어있지 않았을 때 Claude Code가 깔끔하게 멈추고 보고. 가정을 명시하지 않았다면 Claude Code가 추측 기반 복구를 시도했을 가능성이 있음.

지시문에 "사전 상태 가정" 섹션을 두는 것은 실패를 실패답게 멈추게 하는 장치다. 모든 지시문에 이 섹션을 루틴으로 넣으면 예상 밖 상황이 조용히 지나가지 않는다.

## 여담 / 느낀 점

첫 세션은 예상보다 길었다. 게임 컨셉 결정과 아키텍처 논의에 많은 시간이 쓰였지만, 그 덕에 "왜 이렇게 세팅하는가"가 명확한 상태에서 세팅에 들어갈 수 있었다. 만약 바로 세팅부터 시작했다면 중간에 "근데 이거 C#으로 할까 GDScript로 할까" 같은 결정이 세팅 흐름을 끊었을 것이다.

Claude Code는 내가 기대한 것보다 "지시 준수"와 "경계 인식"이 좋았다. 특히 판단이 애매한 지점에서 멈추고 물어본 것, force push 금지를 지킨 것이 인상적이었다. 이건 지시문 자체가 이런 행동을 유도하도록 설계됐기 때문이기도 하다 — "제약을 명시하면 제약을 지킨다"는 단순한 진실을 다시 확인.

하나 아쉬운 건 원격 push 실패다. GitHub 저장소를 만들 때 기본적으로 `.gitignore` 체크박스가 눌려있는 걸 몰랐다. 다음부터는 저장소 생성 시 "완전히 빈 상태로 만들 것"이 기본 절차가 되어야 한다. 이것도 일종의 "가정 명시"다 — "내가 만든 저장소가 정말 빈 상태인가?"를 확인하는 습관.

다음 세션은 문서 작성이다. 이번 세션은 "실험의 뼈대를 만드는" 작업이었고, 다음은 "뼈대 위에 맥락을 입히는" 작업이 될 것이다.
